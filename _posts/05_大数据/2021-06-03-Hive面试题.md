## 2. Hive 优化与数据倾斜

<a id="q01"></a>
### 2.1. 什么是数据倾斜？它会带来什么问题？

**答：**
数据倾斜通常发生在 MapReduce 或 Hive 的 Reduce 阶段，指的是某些 key 对应的数据量特别大，导致这些数据被集中分配到少数几个 Reducer 上处理。

这样一来，这些 Reducer 的执行时间会远远长于其他 Reducer，形成明显的长尾效应，从而拖慢整个任务的完成时间。

数据倾斜带来的问题主要包括：

* 作业整体执行时间变长
* 集群资源利用不均衡
* 出现长尾任务
* 严重时可能导致某些任务内存溢出或者执行失败

---

<a id="q02"></a>
### 2.2. 实际工作中如何处理数据倾斜？

**答：**
处理数据倾斜，一般要分场景来看。

如果是 **Group By 聚合类倾斜**，比较常见的做法是对热点 key 进行打散，比如在 key 前加随机前缀，让原本集中在一个 Reducer 的数据分散到多个 Reducer 上，后面再做一次聚合汇总。

如果是 **Join 场景的倾斜**，常见思路有：

* 使用 Map Join，把小表广播到 Map 端，减少 Reduce 阶段压力
* 对热点 key 做单独拆分处理
* 让倾斜 key 和非倾斜 key 分开计算

总的来说，最常见也最通用的思路，就是热点 key 打散。

---

<a id="q03"></a>
### 2.3. 当 Hive 任务执行变慢时，一般如何排查？

**答：**
一般从以下三个方向排查：

第一，看上游数据量有没有突然暴增。因为数据量的异常增长，往往会直接导致任务耗时增加。

第二，看执行日志，重点关注有没有数据倾斜、有没有异常长尾任务、有没有数据质量问题，比如空值异常集中、脏数据激增等。

第三，看集群资源有没有变化，比如资源是否变少、并发任务是否增多、队列资源是否被其他任务占用。

整体来说，就是从数据量、执行过程、集群资源这三个方向来定位。

---

<a id="q04"></a>
### 2.4. 如果一个 Hive Join 查询运行效率很低，你会怎么优化？

**答：**
首先会检查左右表的数据量，以及关联键的数据分布情况。一般 Join 耗时很久，核心原因通常还是数据倾斜。

如果是倾斜问题，最常见的优化方式就是对关联键做打散，让原本集中在单个 Reducer 上的数据尽量分散开。

如果右表是小表，还可以考虑转成 Map Join，也就是把小表广播到 Map 端，在 Map 阶段直接完成匹配，减少 Reduce 阶段的压力。

如果在开发阶段就已经能预判这类问题，通常会提前做表设计或 SQL 改写，尽量避免在线上任务中暴露出来。

---

<a id="q05"></a>
### 2.5. Hive 中分区设计不合理会带来哪些问题？如何判断分区是否合理？

**答：**
Hive 中最常见的分区方式是按时间分区，比如按天分区。

如果分区设计不合理，通常会出现两类问题：

第一，如果分区粒度太粗，很多数据会集中在少数几个分区里，这样分区就起不到应有的过滤作用，查询时仍然要扫描大量无关数据。

第二，如果分区粒度太细，会产生大量小文件。因为分区本质上就是 HDFS 上的一种目录组织形式，分区过细就会导致文件数暴增，增加 NameNode 元数据压力，也影响任务性能。

判断分区是否合理，一般要结合三个维度：

* 分区字段是否有实际业务含义
* 数据分布是否均匀
* 查询场景中是否经常按这个字段过滤

一个合理的分区设计，应该既能有效缩小查询范围，又不会制造过多小文件。

---

<a id="q06"></a>
### 2.6. 如果数据倾斜的 key 无法提前预知，该怎么处理？

**答：**
如果不能提前知道 Join 或 Group By 阶段到底哪个 key 会发生倾斜，一般可以从以下三个方向处理：

第一，开启 Hive 自带的倾斜优化能力，比如 Map 侧聚合、Skew Join 等，让执行引擎在运行时自动识别并处理热点 key。

第二，手动调节 Reducer 数量，并根据多次运行情况指定一个合理区间。虽然这不是根治手段，但在某些场景下可以缓解倾斜带来的长尾问题。

第三，如果业务允许，可以在 SQL 层面手动增加打散逻辑，比如对 key 加随机前缀，在后续步骤中再将其恢复回来。这个方法更灵活，但实现复杂度也更高。

---

<a id="q07"></a>
### 2.7. 如果执行计划里出现了全表扫描，你会怎么优化？

**答：**
首先要判断这张表是不是分区表，以及查询条件中是否使用了分区字段进行过滤。

如果是分区表，并且查询中使用了分区字段过滤，那么就有优化空间。这个时候要进一步检查过滤条件的位置是否合理，尤其是在 Join 场景下，不同写法会影响分区裁剪是否生效。

例如在 Left Join 场景下，左表的过滤条件一般放在 `where` 中，右表的过滤条件一般放在 `on` 条件中，这样更容易让执行引擎做正确的过滤和裁剪。

如果是非分区表，或者分区字段根本没有参与过滤，那么无论如何都避免不了全表扫描。这种情况下，能做的优化主要是把过滤条件尽可能提前，让后续 Join、Group By 等操作处理更小的结果集，从而提高整体性能。

---

<a id="q08"></a>
### 2.8. 非分区表在 where 中加过滤条件，执行时是不是仍然会全表扫描？

**答：**
是的。

如果是非分区表，即使在 `where` 条件中加了过滤，Hive 仍然需要先把整张表扫描一遍，再对扫描出来的数据做过滤。

也就是说，全表扫描本身无法避免，但提前过滤仍然是有价值的。因为虽然底层扫描没减少，但后续参与 Join、聚合、排序的数据量会变小，从而减少下游计算开销，提升整体性能。

---

<a id="q09"></a>
### 2.9. 为什么说在 Join 场景下要尽量提前过滤？

**答：**
因为即使某些场景无法避免底层的全表扫描，提前过滤仍然可以让后续操作处理更小的数据集。

比如在 Join、Group By、Order By 等重计算场景中，参与计算的数据量越小，Shuffle 开销、网络传输开销、Reduce 压力通常就越小。

所以提前过滤的核心价值不一定是避免扫描，而是减少后续算子的输入规模。

---

<a id="q10"></a>
### 2.10. 大表 Join 小表时，Map Join 是如何优化性能的？

**答：**
当右表足够小的时候，可以开启 Map Join。它的核心思路是把小表加载到每个 Map Task 的内存中，让大表在 Map 阶段直接和小表完成匹配，而不再经过 Reduce 阶段。

这样做的好处是：

* 避免了大规模 Shuffle
* 减少了网络传输开销
* 减轻了 Reduce 阶段压力
* 通常能显著提升 Join 性能

所以在大表 Join 小表的场景里，只要小表足够小，Map Join 往往是优先考虑的方案。

## 3. Hive 原理与优化

<a id="q11"></a>
### 3.1. Hive 的底层执行原理是什么？

**答：**
Hive 本身不是计算引擎，它更像 SQL 到分布式执行计划的转换层。

一条 Hive SQL 会先被解析、优化，再翻译成底层执行引擎可运行的任务计划，底层可能是 MR、Tez 或 Spark。

所以 Hive 的本质是：`SQL 语义层 + 优化层 + 底层执行引擎适配层`。

---

<a id="q12"></a>
### 3.2. 一条 Hive SQL 从提交到执行，中间经历了哪些阶段？

**答：**
通常可以概括成：

1. 词法和语法解析
2. 语义分析，校验表、字段、分区等元数据
3. 逻辑优化，比如谓词下推、列裁剪、Join 优化
4. 生成物理执行计划
5. 提交到底层引擎执行

所以从用户看是一条 SQL，从系统看其实经历了“解析 -> 优化 -> 编排 -> 执行”完整链路。

---

<a id="q13"></a>
### 3.3. Hive 的内部表和外部表有什么区别？

**答：**
两者最核心的区别在于：Hive 是否把底层数据文件也一并当成自己管理的生命周期对象。

* **内部表**：Hive 更像“既管元数据，也管数据文件”
* **外部表**：Hive 主要管元数据，底层文件通常由外部系统或用户自己负责

所以选哪种，关键看你要不要把数据文件生命周期交给 Hive。

---

<a id="q14"></a>
### 3.4. drop table 时，内部表和外部表的行为有什么不同？

**答：**
内部表 `drop table` 时，通常会删除元数据和底层数据文件。

外部表 `drop table` 时，一般只删除 Hive 元数据，底层文件仍然保留。

这是面试里最常被拿来区分内部表和外部表的一点。

---

<a id="q15"></a>
### 3.5. Hive 分区表和分桶表有什么区别？

**答：**
分区表是按某个字段把数据分到不同目录里，核心价值是减少扫描范围。

分桶表是把同一张表的数据按 bucket key 分散到固定数量文件中，核心价值是优化某些 Join、采样和数据组织场景。

一句话区分：

* 分区偏“目录级裁剪”
* 分桶偏“文件内部分布优化”

---

<a id="q16"></a>
### 3.6. 分桶的作用到底是什么？生产中常用吗？

**答：**
分桶的价值主要在于让数据在某个 key 维度上分布更可控，有助于：

* 某些 Join 优化
* 抽样分析
* 减少局部数据倾斜

但生产里没有分区那么普遍，因为它对写入规范、bucket 数设计和后续使用方式要求更高，用不好收益不明显。

---

<a id="q17"></a>
### 3.7. 动态分区和静态分区有什么区别？

**答：**
静态分区是在写入时明确指定分区值，动态分区则是在运行时根据数据内容自动决定落到哪个分区。

静态分区更可控，动态分区更灵活，但也更容易因为分区值过多而制造大量小文件。

所以动态分区好用，但不能放任它无限长。

---

<a id="q18"></a>
### 3.8. 动态分区为什么容易产生小文件？

**答：**
因为动态分区会把不同分区值的数据自动拆到不同目录和文件中。

如果分区值很多、每个分区数据量又不大，就很容易出现“每个分区都只有一点点数据，却生成一个或多个文件”的情况。

所以动态分区的小文件问题，本质上是“分区数量多、单分区数据稀疏”造成的。

---

<a id="q19"></a>
### 3.9. ORC 和 Parquet 为什么更适合数仓？

**答：**
因为它们都是面向分析场景优化过的列式存储格式。

相比普通文本格式，它们通常具备：

* 列裁剪能力
* 更好的压缩率
* 更快的扫描效率
* 更丰富的统计信息，利于优化器做判断

所以数仓里常见“明细表落 ORC / Parquet，少用纯文本”。

---

<a id="q20"></a>
### 3.10. 列式存储为什么适合分析场景？

**答：**
分析查询往往只会用到部分字段，而不是整行所有字段。

列式存储的优势就在于：只读需要的列，减少无关 IO；同列数据类型相近，也更容易压缩。

所以对大表聚合、过滤、统计类查询，列式存储通常比行式更高效。

---

<a id="q21"></a>
### 3.11. Snappy、Gzip、LZO 等压缩方式有什么区别？

**答：**
可以从压缩率和解压速度两个维度看：

* **Gzip**：压缩率高，但 CPU 开销较大
* **Snappy**：压缩率不是最高，但解压很快，分析场景常用
* **LZO**：速度也较快，但生态使用没有 Snappy 那么常见

所以生产里很多数仓场景会偏向 Snappy，因为综合吞吐更平衡。

---

<a id="q22"></a>
### 3.12. Hive 为什么查询慢？

**答：**
Hive 查询慢，通常不是单一原因，而是几类问题叠加：

* 扫描数据太多，比如没分区裁剪
* Shuffle 太重，比如 Join / Group By / distinct
* 小文件过多
* SQL 写法不合理，导致执行计划复杂
* 集群资源不足或排队严重

所以 Hive 慢的排查思路，一定要同时看 SQL、数据和资源。

---

<a id="q23"></a>
### 3.13. Hive SQL 优化一般从哪些层面入手？

**答：**
通常会从四层看：

* **数据层**：分区、分桶、文件格式、小文件治理
* **SQL 层**：提前过滤、改写 Join、减少重复计算
* **执行层**：Map Join、并行度、引擎选择
* **资源层**：YARN 队列、内存、并发参数

高效优化通常不是只调一个参数，而是这四层一起看。

---

<a id="q24"></a>
### 3.14. distinct 和 group by 有什么区别？哪个更容易出问题？

**答：**
从语义上看，`distinct` 是去重，`group by` 是分组聚合。

但在大数据场景下，两者都可能触发重 Shuffle。相对来说，`count(distinct)` 和多字段 `distinct` 更容易在性能上出问题，因为它们对去重范围和数据分布更敏感。

所以不是说谁绝对更差，而是要看实际数据量和 key 分布。

---

<a id="q25"></a>
### 3.15. count(distinct xx) 为什么容易慢？

**答：**
因为它通常要求在全局范围内对某个字段做去重，再统计数量。

这会带来两个开销：

* 大量数据需要按 key 重分发
* 热点值或高基数字段更容易放大 Shuffle 成本

所以 `count(distinct)` 慢，往往慢在全局去重这一步。

---

<a id="q26"></a>
### 3.16. 多个 count(distinct) 为什么性能差？

**答：**
因为多个 distinct 往往意味着系统要维护多套不同去重口径。

这类 SQL 常见问题是：

* 执行计划变复杂
* Shuffle 次数变多
* 中间数据膨胀

所以多个 `count(distinct)` 往往比单个更容易触发复杂 stage 和性能问题。

---

<a id="q27"></a>
### 3.17. Hive 中 order by、sort by、distribute by、cluster by 有什么区别？

**答：**
可以简单记：

* `order by`：全局有序，通常只有一个 Reduce，最贵
* `sort by`：每个 Reduce 内部有序，不保证全局有序
* `distribute by`：决定数据按什么规则分发到不同 Reduce
* `cluster by`：等价于 `distribute by + sort by`，并且字段相同

面试里最关键的一点是：只有 `order by` 保证全局有序。

---

<a id="q28"></a>
### 3.18. left join、right join、inner join、full join 在执行和优化上有什么区别？

**答：**
语义上看，差异在于保留哪一侧或两侧数据。

优化上最重要的区别是：

* `inner join` 通常更灵活，过滤和重排空间更大
* 外连接要保留某侧空值补齐结果，很多过滤下推和重排需要更谨慎
* `full join` 一般最重，优化空间也相对最小

所以从性能角度看，通常 `inner join` 最容易优化。

---

<a id="q29"></a>
### 3.19. 为什么 map join 更快？它的限制是什么？

**答：**
map join 更快，是因为它把小表广播到 Map 端，避免了大表和小表在 Reduce 端做大规模 Shuffle。

它的限制主要是：

* 小表必须足够小，能被各个 Map 节点加载
* 不适合多个大表 Join
* 对内存更敏感

所以 map join 的本质是“用空间换 Shuffle 成本”。

---

<a id="q30"></a>
### 3.20. Hive 如何做谓词下推？

**答：**
谓词下推就是尽可能把过滤条件提前到更靠近数据源的位置执行。

这样做的价值是：

* 少读无关数据
* 少传输无关数据
* 让后续 Join / 聚合处理更小的数据集

所以谓词下推的核心不是语法技巧，而是减少无效扫描和无效计算。

---

<a id="q31"></a>
### 3.21. 列裁剪是什么？

**答：**
列裁剪就是只读取 SQL 真正需要的那些列，不把整张表所有字段都扫出来。

在 ORC、Parquet 这类列式格式里，这个能力特别重要，因为它可以直接减少底层读取量。

所以列裁剪和谓词下推一样，都是“尽量少读”的优化手段。

---

<a id="q32"></a>
### 3.22. 向量化执行是什么？

**答：**
向量化执行指的是算子不再一行一行处理数据，而是按一批一批的列向量处理。

这样通常能带来：

* 更少的函数调用开销
* 更好的 CPU cache 利用
* 更高的执行吞吐

所以它本质上是执行引擎层面的批处理优化。

---

<a id="q33"></a>
### 3.23. Hive 中小文件过多怎么治理？

**答：**
治理思路通常分三层：

* **源头控制**：减少无意义分区、控制并发输出、合并小批次写入
* **过程优化**：使用 CombineInput、合理设置 Reduce 数量
* **结果治理**：对历史分区做 compaction 或重写合并

所以小文件治理一定要做“源头 + 过程 + 存量”三段治理。

---

<a id="q34"></a>
### 3.24. 什么情况下会触发多个 MR / Tez / Spark stage？

**答：**
一般在存在多个 Shuffle 边界时，就容易拆成多个 stage。

常见触发点包括：

* 多次 Join
* Group By / distinct
* Order By
* 某些子查询或 union 重写

所以 stage 多不一定是错，但通常意味着执行路径更复杂、落盘和网络成本更高。

---

<a id="q35"></a>
### 3.25. Hive on MR、Hive on Tez、Hive on Spark 有什么区别？

**答：**
区别主要在底层执行引擎。

* **Hive on MR**：最传统，稳定但延迟高
* **Hive on Tez**：DAG 化更强，通常比 MR 更高效
* **Hive on Spark**：借助 Spark 的执行能力，适合更复杂场景

所以 Hive 是统一 SQL 层，MR / Tez / Spark 是不同执行底座。

---

<a id="q36"></a>
### 3.26. 为什么有些 SQL 看起来简单，执行计划却很复杂？

**答：**
因为 SQL 的表面写法和底层执行并不是一一对应的。

一个看起来简单的 SQL，底层可能经过了：

* 子查询展开
* Join 重写
* 聚合拆分
* 多阶段 Shuffle

所以判断复杂度不能只看 SQL 行数，更要看它触发了哪些物理执行过程。

---

<a id="q37"></a>
### 3.27. explain 执行计划主要看什么？

**答：**
优先看这些：

* 有没有分区裁剪
* 有没有全表扫描
* Join 类型是什么，是否触发 map join
* stage 数量多不多
* 哪些算子会触发 Shuffle

也就是说，explain 最重要的是帮助你判断“数据扫了多少、数据怎么流、重计算发生在哪”。

---

<a id="q38"></a>
### 3.28. 如何通过 explain 判断有没有分区裁剪？

**答：**
通常可以看执行计划里扫描表的分区信息。

如果计划中明确显示只扫描某些目标分区，而不是全分区 / 全表，就说明分区裁剪生效了；如果没有分区过滤信息，或者直接显示全量扫描，就要怀疑裁剪没有生效。

所以重点是看扫描范围有没有被限制住。

---

<a id="q39"></a>
### 3.29. 如何通过 explain 判断有没有 map join？

**答：**
一般可以看执行计划里是否出现 broadcast / map join 相关标识，或者是否省掉了原本需要的大规模 Reduce 端 Join 阶段。

如果小表被广播到 Map 端，计划里通常会体现这是 map-side join，而不是 common join。

面试里不用死记具体文案，关键是知道看“Join 是在 Map 端做，还是在 Reduce 端做”。

---

<a id="q40"></a>
### 3.30. Hive 中 join 顺序会影响性能吗？为什么？

**答：**
会影响。

因为 join 顺序会影响：

* 中间结果大小
* 过滤是否能尽早生效
* 是否有机会触发 map join
* 后续 Shuffle 压力

通常的思路是尽量先做过滤、先缩小数据，再做更重的 Join。
