MySQL 在数据分析工作中并不只是“查数据的工具”，更常见的角色是：做指标统计、做用户行为分析、做报表口径沉淀、做临时分析验证。真正高频的不是某一条单独语法，而是“聚合 + 日期处理 + 窗口函数 + 分层写法”这套组合拳。

这篇文章不再按数据库教材的方式讲 SQL，而是从**数据分析开发的真实场景**出发，整理一套最常用、最容易复用的 MySQL 写法。

---

## 1. 数据分析 SQL 的核心思路

写分析 SQL 时，先不要急着写 `SELECT *`，而是先想清楚下面四件事：

1. **结果粒度是什么**
   是按天、按用户、按订单、按城市，还是按“用户-天”？
2. **指标定义是什么**
   是订单数、支付人数、支付金额，还是留存率、转化率、累计值？
3. **原始明细表有哪些**
   订单表、支付表、用户表、行为日志表，谁是主表，谁是补充维度？
4. **统计口径是否唯一**
   去重逻辑、时间口径、状态口径、是否只看首单，必须提前统一。

一句话总结：

> **分析 SQL 的第一步不是写语法，而是先定义结果表的“行”和“列”。**

## 2. 最常用的基础统计写法

### 2.1. 明细过滤 + 分组聚合

```sql
SELECT
  dt,
  COUNT(*) AS order_cnt,
  SUM(pay_amount) AS pay_amt
FROM orders
WHERE dt >= '2026-03-01'
  AND dt <  '2026-04-01'
  AND order_status = 'paid'
GROUP BY dt
ORDER BY dt;
```

这是最基础也最常见的报表 SQL。

关键点：

- `WHERE` 先过滤明细；
- `GROUP BY` 决定结果粒度；
- 聚合函数负责把明细压成指标。

### 2.2. 条件聚合

```sql
SELECT
  dt,
  COUNT(*) AS total_order_cnt,
  SUM(CASE WHEN pay_status = 'paid' THEN 1 ELSE 0 END) AS paid_order_cnt,
  SUM(CASE WHEN pay_status = 'refund' THEN 1 ELSE 0 END) AS refund_order_cnt,
  SUM(CASE WHEN pay_status = 'paid' THEN pay_amount ELSE 0 END) AS paid_amt
FROM orders
WHERE dt >= '2026-03-01'
  AND dt <  '2026-04-01'
GROUP BY dt
ORDER BY dt;
```

条件聚合是数据分析 SQL 的核心技能之一，适合在一次扫描中输出多个指标。

高频场景：

- 支付订单数、退款订单数、取消订单数；
- 新用户数、老用户数；
- 各渠道订单量、金额、转化人数。

### 2.3. 去重统计

```sql
SELECT
  dt,
  COUNT(DISTINCT user_id) AS dau
FROM user_visit_log
WHERE dt >= '2026-03-01'
  AND dt <  '2026-04-01'
GROUP BY dt;
```

常见指标：

- `COUNT(DISTINCT user_id)`：去重用户数；
- `COUNT(DISTINCT order_id)`：去重订单数；
- `COUNT(DISTINCT CASE WHEN ... THEN user_id END)`：条件去重人数。

例如统计支付用户数：

```sql
SELECT
  dt,
  COUNT(DISTINCT CASE WHEN pay_status = 'paid' THEN user_id END) AS pay_user_cnt
FROM orders
GROUP BY dt;
```

## 3. 多表关联时最容易踩的坑

### 3.1. 维度补充

```sql
SELECT
  a.user_id,
  a.order_id,
  a.pay_amount,
  b.city_name,
  b.register_channel
FROM orders a
LEFT JOIN user_profile b
  ON a.user_id = b.user_id;
```

这类 SQL 的目标通常不是“过滤”，而是“补维度”。

### 3.2. `LEFT JOIN` 条件写在 `ON` 还是 `WHERE`

```sql
SELECT
  a.user_id,
  b.vip_level
FROM user_profile a
LEFT JOIN user_level b
  ON a.user_id = b.user_id
 AND b.is_active = 1;
```

上面表示：

- 先保留左表全部用户；
- 只关联右表中有效等级数据；
- 右表没匹配上时补 `NULL`。

如果你把 `b.is_active = 1` 写进 `WHERE`，就可能把 `LEFT JOIN` 变成近似 `INNER JOIN`。

### 3.3. 一对多膨胀

```sql
SELECT
  a.user_id,
  SUM(a.pay_amount) AS pay_amt
FROM orders a
LEFT JOIN order_item b
  ON a.order_id = b.order_id
GROUP BY a.user_id;
```

如果一笔订单对应多条商品明细，订单金额会被重复累计。

解决思路：

- 先在子查询中把明细表聚合到订单粒度；
- 再和订单表关联；
- 或者先明确自己到底要订单粒度还是商品粒度。

## 4. 日期函数：报表开发高频写法

日期函数在数据分析里几乎天天会用到，尤其是日报、周报、月报、近 N 天统计、用户生命周期分析。

### 4.1. 获取当前时间

```sql
SELECT
  NOW() AS current_datetime,
  CURDATE() AS current_date,
  CURTIME() AS current_time;
```

- `NOW()`：当前日期时间；
- `CURDATE()`：当前日期；
- `CURTIME()`：当前时间。

### 4.2. 提取年月日

```sql
SELECT
  order_id,
  YEAR(pay_time) AS year_num,
  MONTH(pay_time) AS month_num,
  DAY(pay_time) AS day_num
FROM orders;
```

高频用途：

- 提取自然年、自然月；
- 做年/月/日维度统计；
- 做节假日、工作日分析时拆日期字段。

### 4.3. 日期格式化

```sql
SELECT
  DATE_FORMAT(pay_time, '%Y-%m-%d') AS day_str,
  DATE_FORMAT(pay_time, '%Y-%m') AS month_str,
  SUM(pay_amount) AS pay_amt
FROM orders
GROUP BY DATE_FORMAT(pay_time, '%Y-%m-%d'),
         DATE_FORMAT(pay_time, '%Y-%m');
```

常见格式：

- `%Y-%m-%d`：`2026-04-04`
- `%Y-%m`：`2026-04`
- `%H:%i:%s`：`14:30:00`

注意：

> `DATE_FORMAT(pay_time, '%Y-%m')` 可以用于展示，但如果写在过滤条件中，往往不利于索引利用。

### 4.4. 日期加减

```sql
SELECT
  DATE_ADD('2026-04-01', INTERVAL 7 DAY) AS plus_7_day,
  DATE_SUB('2026-04-01', INTERVAL 1 MONTH) AS minus_1_month;
```

高频场景：

- 近 7 天、近 30 天；
- T+1、T+7、T+30 留存；
- 账期、账单周期分析。

### 4.5. 时间差计算

```sql
SELECT
  order_id,
  TIMESTAMPDIFF(MINUTE, create_time, pay_time) AS pay_minutes,
  DATEDIFF(pay_time, create_time) AS pay_days
FROM orders;
```

- `DATEDIFF()`：按天算差值；
- `TIMESTAMPDIFF()`：支持分钟、小时、天、月等粒度。

例如统计订单支付时长：

```sql
SELECT
  AVG(TIMESTAMPDIFF(MINUTE, create_time, pay_time)) AS avg_pay_minutes
FROM orders
WHERE pay_status = 'paid';
```

### 4.6. 时间过滤推荐写法

推荐写法：

```sql
WHERE pay_time >= '2026-03-01'
  AND pay_time <  '2026-04-01'
```

不推荐写法：

```sql
WHERE DATE_FORMAT(pay_time, '%Y-%m') = '2026-03'
```

原因：

- 闭开区间语义清晰；
- 避免漏掉带时分秒的数据；
- 更容易利用索引。

## 5. 开窗函数：数据分析最值钱的语法之一

窗口函数不会像 `GROUP BY` 那样把多行压成一行，而是在**保留明细行的同时，给每一行增加统计信息**。

这非常适合做：

- 每组 TopN；
- 排名；
- 累计值；
- 环比、同比；
- 前后行为对比；
- 首次行为、最近行为分析。

### 5.1. `ROW_NUMBER()`：每组 TopN

```sql
SELECT *
FROM (
  SELECT
    user_id,
    order_id,
    pay_amount,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY pay_amount DESC
    ) AS rn
  FROM orders
) t
WHERE rn <= 3;
```

用途：

- 每个用户金额最高的 3 笔订单；
- 每个城市销量前 5 的商品；
- 每个渠道转化最高的 10 个活动。

### 5.2. `RANK()` 和 `DENSE_RANK()`

```sql
SELECT
  product_id,
  category_id,
  sales_amt,
  RANK() OVER (PARTITION BY category_id ORDER BY sales_amt DESC) AS rk,
  DENSE_RANK() OVER (PARTITION BY category_id ORDER BY sales_amt DESC) AS drk
FROM product_sales;
```

区别：

- `ROW_NUMBER()`：绝不并列；
- `RANK()`：并列后跳号；
- `DENSE_RANK()`：并列后不跳号。

### 5.3. 累计值

```sql
SELECT
  dt,
  day_amt,
  SUM(day_amt) OVER (
    ORDER BY dt
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cum_amt
FROM daily_sales;
```

常见场景：

- 月累计 GMV；
- 月累计新增用户；
- 活动期间累计转化量。

### 5.4. 前后对比：`LAG()` / `LEAD()`

```sql
SELECT
  dt,
  day_amt,
  LAG(day_amt, 1) OVER (ORDER BY dt) AS prev_day_amt,
  day_amt - LAG(day_amt, 1) OVER (ORDER BY dt) AS diff_amt
FROM daily_sales;
```

用途：

- 环比增长；
- 看今天比昨天多多少；
- 看本次行为与上次行为的间隔。

例如计算日环比：

```sql
SELECT
  dt,
  day_amt,
  ROUND(
    (day_amt - prev_day_amt) / NULLIF(prev_day_amt, 0),
    4
  ) AS day_growth_rate
FROM (
  SELECT
    dt,
    day_amt,
    LAG(day_amt) OVER (ORDER BY dt) AS prev_day_amt
  FROM daily_sales
) t;
```

### 5.5. 用户首次行为与最近行为

```sql
SELECT *
FROM (
  SELECT
    user_id,
    event_time,
    event_type,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY event_time ASC
    ) AS first_rn,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY event_time DESC
    ) AS last_rn
  FROM user_event_log
) t
WHERE first_rn = 1 OR last_rn = 1;
```

这是分析“首登、首单、最近一次活跃、最近一次支付”时非常高频的模板。

## 6. CTE 与分层写法：复杂分析 SQL 的关键

数据分析 SQL 一旦复杂起来，往往不是一层 `SELECT` 能解决的。

推荐把 SQL 分成几层：

1. 取明细；
2. 做清洗或补维度；
3. 做中间指标；
4. 最后输出结果。

### 6.1. 用 CTE 分步表达

```sql
WITH order_base AS (
  SELECT
    user_id,
    order_id,
    pay_time,
    pay_amount,
    DATE_FORMAT(pay_time, '%Y-%m-%d') AS dt
  FROM orders
  WHERE pay_status = 'paid'
),
daily_user_pay AS (
  SELECT
    dt,
    user_id,
    COUNT(*) AS pay_order_cnt,
    SUM(pay_amount) AS pay_amt
  FROM order_base
  GROUP BY dt, user_id
)
SELECT
  dt,
  COUNT(DISTINCT user_id) AS pay_user_cnt,
  SUM(pay_order_cnt) AS pay_order_cnt,
  SUM(pay_amt) AS pay_amt
FROM daily_user_pay
GROUP BY dt
ORDER BY dt;
```

优点：

- 结构清晰；
- 方便调试中间结果；
- 不容易把业务口径写乱。

### 6.2. 分层写法建议

可以把分析 SQL 理解成下面四层：

- `base`：明细层，只做必要过滤；
- `dim`：维度补充层；
- `agg`：指标聚合层；
- `final`：最终展示层。

这类命名习惯在多人协作和复盘 SQL 时特别有用。

## 7. 数据分析中的常用函数补充

### 7.1. 空值处理

```sql
SELECT
  user_id,
  IFNULL(city_name, '未知') AS city_name,
  COALESCE(phone, email, '未留资') AS contact_info
FROM user_profile;
```

- `IFNULL(a, b)`：如果 `a` 为 `NULL`，返回 `b`；
- `COALESCE(a, b, c)`：返回第一个非 `NULL` 的值。

### 7.2. 条件判断

```sql
SELECT
  user_id,
  CASE
    WHEN total_amt >= 10000 THEN '高价值用户'
    WHEN total_amt >= 3000 THEN '中价值用户'
    ELSE '普通用户'
  END AS user_level
FROM user_value_tag;
```

`CASE WHEN` 除了聚合里好用，也非常适合做人群分层、标签分桶。

### 7.3. 字符串处理

```sql
SELECT
  user_id,
  CONCAT(province, '-', city) AS city_full_name,
  CONCAT_WS('-', country, province, city) AS full_region_name,
  SUBSTRING(phone, 1, 3) AS phone_prefix,
  TRIM(user_name) AS clean_user_name
FROM user_profile;
```

常用于：

- 拼接维度字段；
- 清洗字符串；
- 提取编码前缀、渠道前缀。

#### 字符串拼接函数补充

在数据分析场景里，最常用的拼接函数主要有 3 个：

1. `CONCAT(str1, str2, ...)`
   - 直接把多个字符串拼接在一起；
   - **只要任意参数为 `NULL`，整体结果就是 `NULL`**。
2. `CONCAT_WS(separator, str1, str2, ...)`
   - `WS` 的含义是 *with separator*；
   - 自动使用分隔符连接多个字符串；
   - 会跳过 `NULL` 参数（但不会跳过空字符串）。
3. `GROUP_CONCAT(expr)`
   - 把多行值聚合成一个字符串；
   - 常用于“每个用户对应多个标签”这种结果展示。

示例：

```sql
SELECT
  CONCAT('A', '-', 'B') AS ex1,                 -- A-B
  CONCAT('A', '-', NULL) AS ex2,                -- NULL
  CONCAT_WS('-', 'A', NULL, 'B') AS ex3,        -- A-B
  GROUP_CONCAT(tag ORDER BY tag SEPARATOR ',') AS tags
FROM user_tag
GROUP BY user_id;
```

> 实战建议：如果你的字段里可能出现 `NULL`，又不希望拼接结果整体变成 `NULL`，优先使用 `CONCAT_WS`，或先用 `COALESCE` 兜底。

### 7.4. `NULL` 值参与判断的逻辑（非常重要）

SQL 是三值逻辑：`TRUE`、`FALSE`、`UNKNOWN`。当表达式涉及 `NULL` 时，很多比较结果不是 `TRUE/FALSE`，而是 `UNKNOWN`。

在 `WHERE` 中，只有 `TRUE` 会被保留；`FALSE` 和 `UNKNOWN` 都会被过滤掉。

```sql
SELECT
  1 = 1            AS t1,       -- TRUE
  1 = NULL         AS t2,       -- NULL(UNKNOWN)
  NULL = NULL      AS t3,       -- NULL(UNKNOWN)
  NULL <> 1        AS t4;       -- NULL(UNKNOWN)
```

所以要注意：

1. 判断空值必须用 `IS NULL` / `IS NOT NULL`，不要用 `= NULL`。
2. `IN` / `NOT IN` 遇到 `NULL` 需要格外谨慎。

```sql
-- 正确：判断空值
WHERE city_name IS NULL

-- 高风险：子查询结果只要包含 NULL，就可能导致整条 NOT IN 逻辑“失效”
WHERE user_id NOT IN (SELECT inviter_id FROM invite_relation)
```

更稳妥的替代方式通常是 `NOT EXISTS`：

```sql
WHERE NOT EXISTS (
  SELECT 1
  FROM invite_relation r
  WHERE r.inviter_id = u.user_id
)
```

另外，`CASE WHEN` 里也要考虑 `NULL`：

```sql
CASE
  WHEN score >= 60 THEN 'pass'
  WHEN score < 60 THEN 'fail'
  ELSE 'unknown'
END
```

如果 `score` 为 `NULL`，前两条都不会命中，结果走 `ELSE`。

## 8. 常见分析模板

### 8.1. 每日核心指标看板

```sql
SELECT
  dt,
  COUNT(DISTINCT user_id) AS active_user_cnt,
  COUNT(DISTINCT CASE WHEN pay_status = 'paid' THEN user_id END) AS pay_user_cnt,
  COUNT(CASE WHEN pay_status = 'paid' THEN 1 END) AS pay_order_cnt,
  SUM(CASE WHEN pay_status = 'paid' THEN pay_amount ELSE 0 END) AS pay_amt
FROM orders
WHERE dt >= '2026-03-01'
  AND dt <  '2026-04-01'
GROUP BY dt
ORDER BY dt;
```

### 8.2. 新老用户拆分

```sql
WITH first_pay AS (
  SELECT
    user_id,
    MIN(DATE(pay_time)) AS first_pay_dt
  FROM orders
  WHERE pay_status = 'paid'
  GROUP BY user_id
)
SELECT
  DATE(o.pay_time) AS dt,
  COUNT(DISTINCT CASE WHEN f.first_pay_dt = DATE(o.pay_time) THEN o.user_id END) AS new_pay_user_cnt,
  COUNT(DISTINCT CASE WHEN f.first_pay_dt < DATE(o.pay_time) THEN o.user_id END) AS old_pay_user_cnt
FROM orders o
JOIN first_pay f
  ON o.user_id = f.user_id
WHERE o.pay_status = 'paid'
GROUP BY DATE(o.pay_time);
```

### 8.3. 留存分析的基本思路

```sql
WITH register_user AS (
  SELECT
    user_id,
    DATE(register_time) AS register_dt
  FROM user_profile
),
active_log AS (
  SELECT DISTINCT
    user_id,
    DATE(event_time) AS active_dt
  FROM user_event_log
)
SELECT
  r.register_dt,
  COUNT(DISTINCT r.user_id) AS register_user_cnt,
  COUNT(DISTINCT CASE
    WHEN DATEDIFF(a.active_dt, r.register_dt) = 1 THEN r.user_id
  END) AS d1_retention_user_cnt
FROM register_user r
LEFT JOIN active_log a
  ON r.user_id = a.user_id
GROUP BY r.register_dt;
```

### 8.4. 每组 Top1 最新记录

```sql
SELECT *
FROM (
  SELECT
    user_id,
    order_id,
    pay_time,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY pay_time DESC
    ) AS rn
  FROM orders
) t
WHERE rn = 1;
```

这个模板特别适合：

- 取用户最新一笔订单；
- 取商品最近一次价格；
- 取设备最后一次在线状态。

### 8.5. LeetCode 601. 体育馆的人流量

题意简化：在 `Stadium(id, visit_date, people)` 表中，找出**连续至少 3 条**且 `people >= 100` 的记录，并按 `visit_date` 升序输出。

核心思路（窗口函数法）：

1. 先过滤出 `people >= 100` 的记录；
2. 用 `id - ROW_NUMBER()` 构造“连续段分组键”；
3. 按分组键统计每段长度；
4. 取长度 `>= 3` 的分组并回表输出。

```sql
WITH t1 AS (
  SELECT
    id,
    visit_date,
    people,
    id - ROW_NUMBER() OVER (ORDER BY id) AS grp
  FROM Stadium
  WHERE people >= 100
),
t2 AS (
  SELECT
    grp
  FROM t1
  GROUP BY grp
  HAVING COUNT(*) >= 3
)
SELECT
  t1.id,
  t1.visit_date,
  t1.people
FROM t1
JOIN t2
  ON t1.grp = t2.grp
ORDER BY t1.visit_date;
```

说明：

- `id - ROW_NUMBER()` 在连续 `id` 区间内保持常数，因此可作为连续段标识；
- 题目数据里 `id` 是递增主键，这种解法可直接使用；
- 这是“连续区间问题”的通用模板，面试和业务分析都很高频。

## 9. 常见开发技巧与优化习惯

### 9.1. 先定粒度，再写 SQL

很多 SQL 错误本质上不是语法错，而是**粒度错**。

例如：

- 明明要按“用户-天”统计，却直接按用户聚合；
- 明明要订单粒度，却先关联到了商品明细粒度；
- 明明要去重人数，却写成了订单数。

### 9.2. 过滤尽量前置

```sql
WITH base AS (
  SELECT *
  FROM orders
  WHERE pay_time >= '2026-03-01'
    AND pay_time <  '2026-04-01'
    AND pay_status = 'paid'
)
SELECT
  user_id,
  SUM(pay_amount) AS pay_amt
FROM base
GROUP BY user_id;
```

越早过滤，后面的连接、排序、聚合压力越小。

### 9.3. 明确是否需要去重

- 不需要去重时，优先 `UNION ALL`；
- 做人数统计时，明确是否要 `COUNT(DISTINCT user_id)`；
- 做多表关联时，先判断会不会一对多膨胀。

### 9.4. 谨慎在条件里对字段做函数处理

不推荐：

```sql
WHERE DATE_FORMAT(pay_time, '%Y-%m') = '2026-03'
```

推荐：

```sql
WHERE pay_time >= '2026-03-01'
  AND pay_time <  '2026-04-01'
```

### 9.5. 用 `EXPLAIN` 看执行计划

```sql
EXPLAIN
SELECT
  user_id,
  SUM(pay_amount) AS pay_amt
FROM orders
WHERE pay_time >= '2026-03-01'
  AND pay_time <  '2026-04-01'
GROUP BY user_id;
```

检查重点：

- 是否走索引；
- 是否出现全表扫描；
- 是否有额外排序和临时表。

### 9.6. 给复杂 SQL 留出中间层

不要把所有逻辑塞进一层 SQL。

更推荐：

- 第一层处理时间和状态过滤；
- 第二层做维度补充；
- 第三层做用户/订单级指标；
- 第四层做最终汇总。

这是分析 SQL 可维护性的关键。

---

## 10. 总结

数据分析里的 MySQL，真正高频的不是某个孤立语法点，而是这几类能力的组合：

- **聚合统计**：`GROUP BY`、条件聚合、去重统计；
- **日期处理**：时间过滤、格式化、加减、差值；
- **窗口分析**：排名、累计值、前后对比、首末行为；
- **分层表达**：子查询、CTE、多层结果沉淀；
- **开发技巧**：先定粒度、避免膨胀、过滤前置、善用 `EXPLAIN`。

如果你是做数据分析、报表开发、业务指标建设，建议优先反复练习下面几类题型：

1. 日报/月报统计；
2. 新老用户拆分；
3. 每组 TopN；
4. 累计值和环比；
5. 留存与首单分析。

这些场景练熟之后，大多数业务分析 SQL 都能比较顺手地写出来。
