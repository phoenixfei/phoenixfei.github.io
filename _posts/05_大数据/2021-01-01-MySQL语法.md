一、刷题最常用的语法

1）GROUP BY + HAVING

这是聚合题的核心。

典型用途
	•	统计次数 / 求和 / 平均值
	•	找“买全所有商品的人”
	•	找重复数据 / 异常数据
	•	分组后过滤

模板

SELECT col1, COUNT(*) AS cnt
FROM t
GROUP BY col1
HAVING COUNT(*) > 1;

高频考法
	•	找重复邮箱
	•	找访问次数最多的人
	•	找满足某种聚合条件的用户

面试会问
	•	WHERE 和 HAVING 的区别
	•	为什么聚合条件一般写在 HAVING

⸻

2）CASE WHEN

这是刷题里非常高频的“条件映射器”。

典型用途
	•	条件计数
	•	条件求和
	•	标签分类
	•	自定义排序

模板

SELECT
    user_id,
    SUM(CASE WHEN status = 'paid' THEN 1 ELSE 0 END) AS paid_cnt
FROM orders
GROUP BY user_id;

常见变形

COUNT(CASE WHEN status = 'paid' THEN 1 END)

不过在 MySQL 里我更建议写 SUM(CASE WHEN ... THEN 1 ELSE 0 END)，可读性更稳。

面试会问
	•	CASE WHEN 能不能写在 SELECT / WHERE / ORDER BY / HAVING
	•	条件聚合怎么实现

⸻

3）JOIN

刷题里 LEFT JOIN 和 INNER JOIN 最常见。

重点不是语法，而是三个能力
	•	会不会选 join 类型
	•	能不能处理一对多导致的数据膨胀
	•	知不知道 ON 和 WHERE 对 LEFT JOIN 的影响

高频模板

SELECT a.id, b.name
FROM A a
LEFT JOIN B b
  ON a.id = b.id;

高频考点

LEFT JOIN + WHERE

SELECT *
FROM A a
LEFT JOIN B b
  ON a.id = b.id
WHERE b.id IS NULL;

这是典型的“找 A 中有、B 中没有”的写法。

ON 和 WHERE 的区别

-- 条件写在 ON
LEFT JOIN B b ON a.id = b.id AND b.flag = 1

-- 条件写在 WHERE
LEFT JOIN B b ON a.id = b.id
WHERE b.flag = 1

第二种很容易把 LEFT JOIN 实际变成近似 INNER JOIN。

面试会问
	•	left join 的执行效果
	•	on 过滤和 where 过滤区别
	•	join 后重复数据怎么处理
	•	join 与 union all 的区别

⸻

4）子查询：标量子查询 / IN / EXISTS

刷题里很常见，尤其是集合判断题。

常见模板

SELECT *
FROM t1
WHERE id IN (
    SELECT id FROM t2
);

SELECT *
FROM t1
WHERE EXISTS (
    SELECT 1
    FROM t2
    WHERE t1.id = t2.id
);

经验结论
	•	判断存在性：优先想到 EXISTS
	•	返回固定标量：标量子查询
	•	做全集比较：HAVING ... = (SELECT COUNT(...))

面试会问
	•	相关子查询和非相关子查询区别
	•	EXISTS 和 IN 的差异
	•	子查询能不能优化成 join

⸻

5）UNION ALL

这个在刷题和报表题里都很多。

用途
	•	合并多段结果
	•	拼接不同来源数据
	•	题目要求返回多行不同含义结果

模板

SELECT col FROM t1
UNION ALL
SELECT col FROM t2;

重点
	•	列数一致
	•	类型兼容
	•	UNION ALL 不去重，通常比 UNION 更快

高频坑

如果每段都要 ORDER BY ... LIMIT，要加括号或再包一层子查询，否则很容易报语法或逻辑错误。

⸻

6）窗口函数 OVER (...)

这是 MySQL 8.0 刷题最重要的进阶能力。MySQL 支持窗口函数，窗口函数会对每一行基于其相关行集合做计算，但不会像 GROUP BY 那样把多行压成一行。 ￼

高频函数
	•	ROW_NUMBER()
	•	RANK()
	•	DENSE_RANK()
	•	SUM() OVER
	•	AVG() OVER
	•	LAG()
	•	LEAD()

典型题型

每组取前 1 / 前 N

SELECT *
FROM (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn
    FROM emp
) t
WHERE rn = 1;

连续增长 / 环比

SELECT *,
       LAG(amount) OVER (PARTITION BY user_id ORDER BY dt) AS prev_amount
FROM sales;

面试会问
	•	窗口函数和 GROUP BY 区别
	•	RANK / DENSE_RANK / ROW_NUMBER 区别
	•	为什么 topN per group 用窗口函数更优雅

⸻

7）CTE：WITH

CTE 是单条语句作用域内的命名临时结果集，可以被后续查询多次引用，也支持递归。 ￼

模板

WITH tmp AS (
    SELECT ...
    FROM ...
)
SELECT *
FROM tmp;

适合场景
	•	复杂 SQL 拆步骤
	•	多次复用同一中间结果
	•	配合窗口函数分层写

面试会问
	•	WITH 和子查询 / 临时表 / 视图区别
	•	CTE 会不会一定物化

补一句实战结论：MySQL 优化器对 derived table / view / CTE 会尽量 merge，避免不必要的物化，但不是所有场景都能 merge。 ￼

⸻

8）日期函数

刷题里非常常见。

高频函数
	•	DATE_FORMAT()
	•	DATEDIFF()
	•	TIMESTAMPDIFF()
	•	DATE_ADD() / DATE_SUB()
	•	YEAR() / MONTH() / DAY()

典型题型

按天/月统计

SELECT DATE_FORMAT(create_time, '%Y-%m') AS mon, COUNT(*)
FROM orders
GROUP BY DATE_FORMAT(create_time, '%Y-%m');

日期范围过滤

WHERE create_time >= '2020-02-01'
  AND create_time <  '2020-03-01'

面试会问
	•	为什么不推荐在 WHERE 里对时间列直接 DATE_FORMAT
	•	为什么范围过滤更容易走索引

9）空值处理

刷题常用：
	•	IFNULL(a, b)
	•	COALESCE(a, b, c, ...)

高频用途

例子

SELECT COALESCE(phone, wechat, email, 'unknown') AS contact

二、刷题最该掌握的几个“题型模板”

1. 分组统计类

SELECT key_col, COUNT(*) AS cnt
FROM t
GROUP BY key_col;

2. 分组后过滤类

SELECT key_col
FROM t
GROUP BY key_col
HAVING COUNT(*) >= 3;

3. 条件聚合类

SELECT user_id,
       SUM(CASE WHEN status = 'paid' THEN 1 ELSE 0 END) AS paid_cnt
FROM t
GROUP BY user_id;

4. 查重类

SELECT email
FROM Person
GROUP BY email
HAVING COUNT(*) > 1;

5. 集合覆盖类

SELECT customer_id
FROM Customer
GROUP BY customer_id
HAVING COUNT(DISTINCT product_key) = (
    SELECT COUNT(DISTINCT product_key)
    FROM Product
);

6. 每组 top1 / topN

SELECT *
FROM (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY group_col ORDER BY metric DESC) AS rn
    FROM t
) x
WHERE rn <= 3;

7. 反连接 anti join

SELECT a.*
FROM A a
LEFT JOIN B b
  ON a.id = b.id
WHERE b.id IS NULL;

8. 连续值 / 前后比较

SELECT *,
       LAG(val) OVER (PARTITION BY user_id ORDER BY dt) AS prev_val
FROM t;

一、GROUP BY / HAVING / 聚合相关

1. WHERE 和 HAVING 有什么区别？

回答：

WHERE 是分组前过滤，作用于原始明细数据；
HAVING 是分组后过滤，作用于聚合结果。

比如：

SELECT dept_id, COUNT(*) AS cnt
FROM emp
WHERE status = 'A'
GROUP BY dept_id
HAVING COUNT(*) > 10;

这里：
	•	WHERE status = 'A'：先过滤状态为 A 的记录
	•	HAVING COUNT(*) > 10：再对分组结果过滤，只保留人数大于 10 的部门

一句话记忆：
	•	WHERE：分组前
	•	HAVING：分组后

⸻

2. 为什么聚合条件一般写在 HAVING 里？

回答：

因为聚合函数如 COUNT()、SUM()、AVG() 是在 GROUP BY 之后才计算出来的，所以不能直接在 WHERE 中使用，通常要写在 HAVING 中。

例如：

SELECT customer_id
FROM orders
GROUP BY customer_id
HAVING COUNT(*) >= 3;

这里是先按客户分组，再筛选下单次数大于等于 3 的客户。

⸻

3. COUNT(*)、COUNT(1)、COUNT(col) 有什么区别？

回答：
	•	COUNT(*)：统计总行数，包括 NULL
	•	COUNT(1)：本质上也是统计总行数
	•	COUNT(col)：只统计该列非 NULL 的行数

例如某列有 10 行，其中 2 行是 NULL：
	•	COUNT(*) = 10
	•	COUNT(1) = 10
	•	COUNT(col) = 8

补充：
在 MySQL 里，COUNT(*) 和 COUNT(1) 性能上通常没有本质区别。

⸻

4. COUNT(DISTINCT col) 为什么通常更重？

回答：

因为它不仅要统计，还要先做去重。
相比普通 COUNT(*) 或 COUNT(col)，COUNT(DISTINCT) 往往需要更多排序、哈希或临时去重处理，所以开销更大。

这类语句常见于：

COUNT(DISTINCT product_key)

尤其在大表、分组场景下，代价会比较明显。

⸻

5. DISTINCT 和 GROUP BY 有什么关系？

回答：

两者都可以起到去重效果，但语义不同：
	•	DISTINCT：强调结果去重
	•	GROUP BY：强调按某列分组聚合

例如：

SELECT DISTINCT dept_id FROM emp;

和

SELECT dept_id FROM emp GROUP BY dept_id;

结果可能一样，但 GROUP BY 更适合后续配合聚合函数使用，比如 COUNT、SUM。

二、CASE WHEN 相关

6. CASE WHEN 常用于哪些场景？

回答：

CASE WHEN 最常用于：
	1.	条件计数
	2.	条件求和
	3.	分类打标签
	4.	自定义排序

例如统计支付订单数：

SELECT
    user_id,
    SUM(CASE WHEN status = 'paid' THEN 1 ELSE 0 END) AS paid_cnt
FROM orders
GROUP BY user_id;


⸻

7. CASE WHEN 可以写在哪些位置？

回答：

CASE WHEN 很灵活，可以写在：
	•	SELECT
	•	WHERE
	•	ORDER BY
	•	HAVING

最常见的是写在 SELECT 中做条件聚合，或者写在 ORDER BY 中做自定义排序。

⸻

8. 为什么条件聚合常写成 SUM(CASE WHEN ... THEN 1 ELSE 0 END)？

回答：

因为这种写法语义最清晰，也兼容性好。

比如统计已支付订单数：

SUM(CASE WHEN status = 'paid' THEN 1 ELSE 0 END)

意思是：
	•	满足条件记 1
	•	不满足记 0
	•	最后求和

这样比某些变体写法更稳定、更容易理解。

三、JOIN 相关

9. INNER JOIN 和 LEFT JOIN 的区别是什么？

回答：
	•	INNER JOIN：只保留两边匹配上的数据
	•	LEFT JOIN：保留左表全部数据，右表匹配不上时补 NULL

例如：

SELECT a.id, b.name
FROM A a
LEFT JOIN B b
  ON a.id = b.id;

即使 B 中没有匹配行，A 的数据也会保留。

⸻

10. LEFT JOIN 中，ON 和 WHERE 的过滤有什么区别？

回答：

这是非常高频的面试题。

条件写在 ON 中：

SELECT *
FROM A a
LEFT JOIN B b
  ON a.id = b.id
 AND b.flag = 1;

含义是：
	•	先按 a.id = b.id 关联
	•	再要求右表满足 b.flag = 1
	•	左表数据仍然保留，右表不满足就补 NULL

条件写在 WHERE 中：

SELECT *
FROM A a
LEFT JOIN B b
  ON a.id = b.id
WHERE b.flag = 1;

含义是：
	•	先 LEFT JOIN
	•	再用 WHERE 过滤结果
	•	因为右表为空时 b.flag 也是 NULL，不满足条件
	•	所以很多左表未匹配的数据会被过滤掉

结论：
	•	ON：影响关联过程
	•	WHERE：影响最终结果集

把右表条件写在 WHERE 里，很容易让 LEFT JOIN 看起来像 INNER JOIN。

⸻

11. 为什么 JOIN 后容易出现重复数据？

回答：

因为连接键可能不是一对一，而是一对多或多对多。

例如：
	•	一个用户有多笔订单
	•	一个商品有多条评价

这时一条左表记录可能匹配多条右表记录，结果集自然会膨胀。

面试补充说法：
如果后面再做聚合统计，不注意这种膨胀，就可能导致结果失真。

⸻

12. 如何避免 JOIN 导致的聚合失真？

回答：

常见方法有：
	1.	先在明细层去重
	2.	先对子表聚合后再 join
	3.	明确使用 COUNT(DISTINCT ...)
	4.	确认 join 关系是否一对多

例如先把订单表聚合到用户级，再和用户表关联，通常比直接 join 后再统计更稳。

⸻

13. JOIN 和 UNION ALL 的区别是什么？

回答：
	•	JOIN：按列进行横向拼接，依赖关联条件
	•	UNION ALL：按行进行纵向拼接，要求列数一致、类型兼容

例如：
	•	JOIN 是把用户信息和订单信息拼在一行
	•	UNION ALL 是把两个查询结果上下合并

四、子查询 / IN / EXISTS 相关

14. 什么是相关子查询和非相关子查询？

回答：
	•	非相关子查询：子查询不依赖外层查询，可以单独执行
	•	相关子查询：子查询依赖外层查询的字段，通常会随着外层行变化而执行

非相关子查询示例：

SELECT *
FROM emp
WHERE dept_id IN (
    SELECT dept_id FROM dept
);

相关子查询示例：

SELECT *
FROM emp e1
WHERE salary > (
    SELECT AVG(salary)
    FROM emp e2
    WHERE e2.dept_id = e1.dept_id
);

这里子查询依赖外层的 e1.dept_id。

⸻

15. EXISTS 和 IN 有什么区别？

回答：

语义上都能做集合判断，但侧重点不同：
	•	IN：判断某值是否在某个结果集中
	•	EXISTS：判断子查询是否存在结果

例如：

WHERE id IN (SELECT id FROM t2)

和

WHERE EXISTS (
    SELECT 1
    FROM t2
    WHERE t1.id = t2.id
)

很多场景下两者可以互相改写。

面试表达：
如果目的是判断“是否存在匹配行”，通常优先想到 EXISTS。

⸻

16. 子查询能不能改写成 JOIN？

回答：

很多时候可以，但不一定必须改。

关键看：
	1.	改写后是否更直观
	2.	是否更容易让优化器处理
	3.	是否会引入重复行问题

所以面试里不要机械回答“join 一定更快”，而是要结合场景分析。

⸻

17. WHERE t.cnt = (SELECT COUNT(...) ...) 这种写法常见吗？

回答：

很常见，属于典型的“分组后计数 = 全集计数”写法，常用于判断某个实体是否覆盖了全集，比如：
	•	买全所有商品的用户
	•	选全所有课程的学生

例如：

HAVING COUNT(DISTINCT product_key) = (
    SELECT COUNT(DISTINCT product_key)
    FROM Product
)

这类题在刷题和面试里都很高频。

⸻

18. 这种写法能优化吗？

回答：

通常可以优化成直接用 HAVING，避免多包一层子查询。

例如原来是：

SELECT customer_id
FROM (
    SELECT customer_id, COUNT(DISTINCT product_key) AS cnt
    FROM Customer
    GROUP BY customer_id
) t
WHERE t.cnt = (SELECT COUNT(*) FROM Product);

可以改成：

SELECT customer_id
FROM Customer
GROUP BY customer_id
HAVING COUNT(DISTINCT product_key) = (
    SELECT COUNT(DISTINCT product_key)
    FROM Product
);

这种写法更自然、更常见。

⸻

⸻

五、UNION ALL 相关

19. UNION 和 UNION ALL 的区别是什么？

回答：
	•	UNION：合并结果并去重
	•	UNION ALL：合并结果但不去重

所以：
	•	明确不需要去重时，优先用 UNION ALL
	•	它通常比 UNION 性能更好

⸻

20. UNION ALL 使用时有什么要求？

回答：

主要有两个要求：
	1.	各个 SELECT 的列数一致
	2.	对应列的数据类型兼容

另外，结果列名通常以第一个 SELECT 为准。

⸻

21. 为什么 UNION ALL 两边单独写 ORDER BY ... LIMIT 容易报错？

回答：

因为 MySQL 中 UNION ALL 连接的每个分支如果要单独排序和限制行数，通常需要：
	•	用括号包住每个分支
	•	或者再套一层子查询

否则语法或执行逻辑会冲突。

⸻

⸻

六、窗口函数相关

22. 窗口函数和 GROUP BY 的区别是什么？

回答：
	•	GROUP BY：会把多行聚合成一行
	•	窗口函数：不会丢失明细行，而是在每一行上计算组内指标

例如：

ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC)

可以给每个部门内部员工按工资排名，但不会把员工明细压缩掉。

⸻

23. ROW_NUMBER、RANK、DENSE_RANK 有什么区别？

回答：

假设分数排序结果是：100，90，90，80

ROW_NUMBER()
编号连续，不管并列：
	•	1
	•	2
	•	3
	•	4

RANK()
并列名次相同，但会跳号：
	•	1
	•	2
	•	2
	•	4

DENSE_RANK()
并列名次相同，但不跳号：
	•	1
	•	2
	•	2
	•	3

⸻

24. 为什么“每组 TopN”常用窗口函数？

回答：

因为窗口函数表达最自然，代码也最清晰。

典型写法：

SELECT *
FROM (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn
    FROM emp
) t
WHERE rn <= 3;

这类题如果不用窗口函数，往往要写相关子查询、自连接，逻辑会更复杂。

⸻

25. LAG 和 LEAD 有什么用？

回答：

用于取当前行的前一行或后一行数据，常用于：
	•	环比
	•	连续天数分析
	•	相邻记录比较

例如：

LAG(amount) OVER (PARTITION BY user_id ORDER BY dt)

表示取同一用户上一天的金额。

⸻

⸻

七、CTE / WITH 相关

26. WITH 是什么？有什么用？

回答：

WITH 就是 CTE，公共表表达式。
可以理解为给一个子查询结果起名字，用于后续 SQL 引用。

例如：

WITH tmp AS (
    SELECT ...
)
SELECT * FROM tmp;

它的好处是：
	1.	提高可读性
	2.	适合拆分复杂 SQL
	3.	中间结果可复用

⸻

27. WITH 和子查询、临时表、视图有什么区别？

回答：

可以从作用域和生命周期理解：

WITH
	•	只在当前 SQL 语句中有效
	•	适合拆分复杂查询

子查询
	•	一般直接写在 FROM、WHERE 等位置
	•	可读性有时不如 CTE

临时表
	•	需要显式创建
	•	可在当前会话中复用多次

视图
	•	是数据库对象
	•	可长期复用

一句话：
	•	WITH：单条 SQL 内的命名中间结果
	•	临时表：会话级中间表
	•	视图：持久化逻辑表

⸻

28. CTE 一定会物化吗？

回答：

不一定。
在 MySQL 中，优化器会根据场景判断是否进行合并或物化，不是说用了 WITH 就一定会落成临时结果。

面试中回答到这个层次就够了：
CTE 主要是逻辑表达工具，是否物化由优化器决定。

⸻

八、日期函数相关

29. 为什么不推荐在 WHERE 中对时间列直接做函数处理？

回答：

因为这样通常不利于索引使用。

例如：

WHERE DATE_FORMAT(create_time, '%Y-%m') = '2020-02'

这是对字段做函数处理，很多情况下索引无法高效利用。

更推荐写成范围过滤：

WHERE create_time >= '2020-02-01'
  AND create_time <  '2020-03-01'

这样通常更利于走索引。

⸻

30. 按月统计和按月过滤分别怎么写更合理？

回答：

按月统计
可以用：

SELECT DATE_FORMAT(create_time, '%Y-%m') AS mon, COUNT(*)
FROM orders
GROUP BY DATE_FORMAT(create_time, '%Y-%m');

按月过滤
更推荐范围写法：

WHERE create_time >= '2020-02-01'
  AND create_time <  '2020-03-01'

面试里通常会追问：
统计展示可以格式化，过滤条件最好写范围。

⸻

九、空值处理相关

31. IFNULL 和 COALESCE 有什么区别？

回答：
	•	IFNULL(a, b)：两个参数，a 为空就返回 b
	•	COALESCE(a, b, c, ...)：多个参数，返回第一个非 NULL 的值

因此：
	•	IFNULL：二选一
	•	COALESCE：多选一

例如：

COALESCE(phone, wechat, email, 'unknown')

表示依次取第一个非空联系方式。

⸻

32. COALESCE 怎么记忆？

回答：

可以记成：

从左到右，谁先不是 NULL，就返回谁。

它本质上就是一个“依次兜底”的函数。

⸻

33. NULLIF 的作用是什么？

回答：

NULLIF(a, b) 的含义是：
	•	如果 a = b，返回 NULL
	•	否则返回 a

常用于避免除零等场景。

例如：

amount / NULLIF(cnt, 0)

如果 cnt = 0，就返回 NULL，避免报错。

十、执行与优化意识相关

34. SQL 的逻辑执行顺序你怎么理解？

回答：

从逻辑上一般可理解为：
	1.	FROM / JOIN
	2.	WHERE
	3.	GROUP BY
	4.	HAVING
	5.	SELECT
	6.	ORDER BY
	7.	LIMIT

不一定要背到极致，但要知道：
	•	WHERE 在分组前
	•	HAVING 在分组后
	•	ORDER BY 是最后排序

这对理解 SQL 语义非常重要。

⸻

35. 为什么说过滤条件要尽量前置？

回答：

因为越早过滤，参与后续 join、group by、sort 的数据量就越少，通常执行代价也越低。

例如能在明细层过滤的条件，就尽量不要拖到最外层再过滤。

⸻

36. 为什么说“明确不需要去重时优先 UNION ALL”？

回答：

因为 UNION 需要额外做去重，而 UNION ALL 只是直接拼接结果，所以通常更高效。

所以在业务上如果重复数据本来就应该保留，或者你确定结果不会重复，就优先考虑 UNION ALL。

⸻

37. 在索引列上套函数为什么不推荐？

回答：

因为这会让数据库难以直接利用原有索引结构进行快速查找，容易退化成全表扫描或低效扫描。

典型例子就是对日期列做 DATE_FORMAT()、YEAR() 等函数后再过滤。

⸻

38. 什么是 join 膨胀？为什么危险？

回答：

join 膨胀是指一对多或多对多连接后，结果行数变多。

危险在于：
	•	会增加计算量
	•	会让后续聚合结果失真
	•	会导致统计重复

所以面试里如果问你 SQL 为什么算多了，很多时候本质就是 join 膨胀。

⸻

十一、适合最后总结的一段面试回答

如果面试官让你总结“你觉得 MySQL 刷题和面试最重要的语法点是什么”，可以这样答：

MySQL 刷题和面试里，最核心的语法主要是聚合、连接、子查询、条件表达式、窗口函数和 CTE。
聚合部分重点掌握 GROUP BY + HAVING、COUNT(DISTINCT) 和 CASE WHEN 条件聚合；
连接部分重点掌握 INNER JOIN、LEFT JOIN 以及 ON 和 WHERE 的过滤差异；
进阶部分要会窗口函数，比如 ROW_NUMBER、RANK、LAG/LEAD，以及 WITH 拆分复杂 SQL。
此外，面试中还会重点考察 SQL 语义理解和优化意识，比如分组前后过滤区别、避免 join 膨胀、尽量使用范围过滤时间字段、明确不去重时使用 UNION ALL 等。
