MySQL 是数据分析、报表开发和业务系统中最常见的关系型数据库之一。本文围绕“高频语法 + 常见问题解析”展开，帮助你快速形成可落地的 SQL 写作思路。

---

## 1. MySQL 语法体系概览

在日常开发中，可以把 MySQL 常用语法分为以下 4 类：

1. **查询与过滤**：`SELECT`、`WHERE`、`ORDER BY`、`LIMIT`。
2. **聚合与分组**：`GROUP BY`、`HAVING`、`COUNT/SUM/AVG`。
3. **多表与集合**：`JOIN`、`UNION ALL`、`EXISTS/IN`。
4. **分析增强**：窗口函数（`OVER`）、CTE（`WITH`）、日期与空值函数。

![MySQL Logo](https://upload.wikimedia.org/wikipedia/en/d/dd/MySQL_logo.svg)

建议学习顺序：

- 先掌握单表查询与分组聚合；
- 再掌握多表关联和子查询改写；
- 最后系统学习窗口函数与 CTE。

## 2. 核心语法：从统计到分析

### 2.1. 分组统计与分组过滤

```sql
SELECT dept_id, COUNT(*) AS emp_cnt
FROM emp
WHERE status = 'active'
GROUP BY dept_id
HAVING COUNT(*) >= 5
ORDER BY emp_cnt DESC;
```

- `WHERE`：分组前过滤明细行。
- `HAVING`：分组后过滤聚合结果。

### 2.2. 条件聚合（CASE WHEN）

```sql
SELECT
  user_id,
  SUM(CASE WHEN status = 'paid' THEN 1 ELSE 0 END) AS paid_cnt,
  SUM(CASE WHEN status = 'refund' THEN 1 ELSE 0 END) AS refund_cnt
FROM orders
GROUP BY user_id;
```

该写法在报表统计中非常常见，语义清晰且可维护性高。

### 2.3. 多表连接（JOIN）

```sql
SELECT a.id, a.name, b.level
FROM user_profile a
LEFT JOIN user_level b
  ON a.id = b.user_id;
```

重点理解：

- `INNER JOIN` 仅保留匹配行；
- `LEFT JOIN` 保留左表全部行，右表不匹配时补 `NULL`；
- 右表过滤条件写在 `WHERE` 中时，可能改变 `LEFT JOIN` 的语义。

## 3. 进阶语法：子查询、窗口函数与 CTE

### 3.1. 子查询与存在性判断

```sql
SELECT *
FROM customer c
WHERE EXISTS (
  SELECT 1
  FROM orders o
  WHERE o.customer_id = c.id
);
```

适用于“是否存在相关记录”的场景。

### 3.2. 窗口函数：组内排序与前后对比

```sql
SELECT *
FROM (
  SELECT
    emp_id,
    dept_id,
    salary,
    ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn
  FROM emp
) t
WHERE rn <= 3;
```

该模式可用于“每组 TopN”问题。

### 3.3. CTE：拆解复杂 SQL

```sql
WITH monthly_sales AS (
  SELECT DATE_FORMAT(pay_time, '%Y-%m') AS mon, SUM(amount) AS total_amt
  FROM payments
  GROUP BY DATE_FORMAT(pay_time, '%Y-%m')
)
SELECT *
FROM monthly_sales
ORDER BY mon;
```

CTE 的优势是结构清晰，适合分步骤表达复杂逻辑。

## 4. 常见问题解析（博客化整理）

以下问题并非“答题模板”，而是实际写 SQL 时最容易踩坑的知识点。

### 4.1. `WHERE` 与 `HAVING` 应如何选择

- 过滤明细记录时使用 `WHERE`；
- 过滤聚合结果时使用 `HAVING`；
- 二者可组合使用，且通常先 `WHERE` 后 `GROUP BY` 再 `HAVING`。

### 4.2. `COUNT(*)`、`COUNT(1)`、`COUNT(col)` 的区别

1. `COUNT(*)`：统计总行数。
2. `COUNT(1)`：本质上也是统计总行数。
3. `COUNT(col)`：仅统计该列非 `NULL` 的行数。

### 4.3. 为什么建议优先 `UNION ALL`

- `UNION` 需要额外去重，开销通常更高；
- 若业务不要求去重，`UNION ALL` 更直接、性能更稳定。

### 4.4. 为什么时间过滤建议写“闭开区间”

推荐：

```sql
WHERE create_time >= '2026-03-01'
  AND create_time <  '2026-04-01'
```

不建议在过滤条件中直接对列做函数处理（例如 `DATE_FORMAT(create_time, ...)`），这类写法通常不利于索引利用。

## 5. 实战写作建议与优化习惯

可以按下面的步骤构建 SQL：

1. **先确定结果粒度**：按用户、按订单、按天还是按月。
2. **再做明细过滤**：优先在 `WHERE` 降低数据量。
3. **然后分组/连接**：明确是否会出现一对多膨胀。
4. **最后做结果整形**：排序、限制返回行数、字段别名优化。

常见优化习惯：

- 明确不去重时使用 `UNION ALL`；
- 谨慎在索引列上套函数；
- 对复杂查询使用 CTE 分层表达；
- 对关键 SQL 使用 `EXPLAIN` 检查执行计划。

```sql
EXPLAIN
SELECT customer_id, COUNT(*) AS order_cnt
FROM orders
WHERE pay_time >= '2026-01-01'
  AND pay_time <  '2026-02-01'
GROUP BY customer_id;
```

## 6. 总结

本文围绕 MySQL 的核心语法、进阶能力和常见问题给出了一套统一的写作框架：

- **基础层**：过滤、分组、聚合；
- **连接层**：`JOIN`、子查询、集合合并；
- **分析层**：窗口函数、CTE、日期与空值处理；
- **工程层**：可读性、可维护性与执行效率并重。

如果你在学习或工作中需要处理复杂统计 SQL，建议从“分组聚合 + 条件聚合 + 每组 TopN”三类场景反复练习，通常可以覆盖大多数业务分析需求。
