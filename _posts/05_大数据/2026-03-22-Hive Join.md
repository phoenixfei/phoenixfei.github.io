Hive 官方和相关设计文档里，常见 Join 优化路径就包括：把 CommonJoin 转成 MapJoin、再进一步转成 BucketJoin、SMBJoin，或者针对数据倾斜转成 SkewJoin。

---

# 1. Common Join / Reduce-Side Join

这是一类 **最基础、最通用** 的 Join。

## 怎么执行

两边表的数据先经过 Map：

* 按 join key 输出
* 然后进入 shuffle
* 相同 key 的数据被分发到同一个 reducer
* reducer 里完成 join

也就是说：

**Join 真正发生在 Reduce 端。**

## 作用

它适合：

* 两边都是大表
* 没有明显“小表可广播”
* 数据也没有提前按 bucket/sort 准备好

## 优点

* 通用性最强
* 不要求小表放得下内存
* 不要求表预先分桶排序

## 缺点

* 要经过 **shuffle**
* 网络传输、排序、磁盘落地开销大
* 最怕热点 key，容易发生数据倾斜

## 和 skew join 的关系

**Skew join 本质上是对 common join 的优化。**
当 common join 遇到热点 key 时，某个 reducer 会特别慢，skew join 就会把“热点 key”单独拆出来处理。Hive 配置说明中明确提到：会把 skewed keys 和非 skewed keys 拆成不同计划，再 union 结果；偏斜 key 的 join 通常会单独走 map-side join 路径。([hive.apache.org][2])

# 2. Map Join / Broadcast Join

这是最常见的优化 Join。

Hive 文档说明，当一侧表能放进内存时，优化器会把这一侧加载成哈希表，只扫描大表，从而自动选择 map join。([hive.apache.org][3])

## 怎么执行

* 选一张 **小表**
* 小表先被加载到每个 mapper 的内存中，通常构造成 hash table
* 大表正常被 mapper 扫描
* 在 **Map 端直接完成 join**
* 不需要大规模 shuffle

## 作用

适合：

* **大表 join 小表**
* 典型维表 join 事实表
* 星型模型场景

## 优点

* 不走 reduce join 主流程
* 大幅减少 shuffle
* 性能通常比 reduce-side join 好很多

## 缺点

* 小表必须足够小，能放进内存
* 小表过大时会 OOM 或退化
* 对大表 join 大表无能为力

## 和 skew join 的区别

* **map join** 解决的是：一边小，可以广播
* **skew join** 解决的是：key 分布不均，某些 key 太热

所以两者解决的问题根本不同。

---

# 3. Bucket Map Join

这是在 map join 基础上的进一步优化。

Hive 的优化链里就包含从 MapJoin 转 BucketJoin。([hive.apache.org][4])

## 怎么执行

前提是表已经按 join key 做了 **bucketing（分桶）**。
这样 join 时，不需要把一张表完整广播到所有 mapper，而是：

* 某个大表 bucket
* 只和对应的小表 bucket 去匹配
* mapper 只处理相关 bucket

## 作用

适合：

* 表已经按 join key 分桶
* 希望进一步减少 map join 的无效数据读取和广播成本

## 优点

* 比普通 map join 更省资源
* 利用 bucket 对齐关系，减少数据扫描量

## 缺点

* 要求表建模时就做过 bucketing
* 使用条件比 map join 严格
* 运维和建表规范要求更高

## 和 map join 的区别

* **map join**：小表通常要“整体”广播到每个 mapper
* **bucket map join**：借助分桶，只让对应 bucket 对应 join，更精细

---

# 4. Sort Merge Bucket Join（SMB Join）

这是更进一步的优化，通常用于 **两边都比较大，但都已经按 join key 分桶且排序** 的场景。

Hive 的优化文档里明确包含从 BucketJoin 到 SMBJoin 的转换。([hive.apache.org][4])

## 怎么执行

要求通常更严格：

* 两边表都按 join key 分桶
* 同时按 join key 排序
* bucket 数或 bucket 对应关系满足要求

这样每个 mapper 可以：

* 读取对应 bucket
* 直接按有序流做 merge join

## 作用

适合：

* 两边都是大表
* 数据长期复用
* join key 很固定
* 能接受前期建模成本

## 优点

* 避免或显著减少 shuffle/sort 成本
* 对大表 join 大表特别有价值
* 比普通 reduce-side join 更高效

## 缺点

* 前提最苛刻
* 要求表提前 bucket + sort
* 对数据建模、落表规范要求高

## 和前几种的区别

* **common join**：最通用，但最重
* **map join**：大表小表最优
* **bucket map join**：在 map join 上利用分桶
* **SMB join**：两边大表但都预处理得很好时最强

---

# 5. Skew Join

你已经提到了，我顺手放在体系里对比。

Hive 配置文档说明，skew join 会把倾斜 key 和普通 key 分开处理，再 union 结果；倾斜 key 在合适条件下会走 map-side join。([hive.apache.org][2])

## 它解决什么

不是单纯“让 join 更快”，而是解决：

* 某些 key 数据量远大于其他 key
* reducer 负载极不均衡
* 长尾任务严重

## 核心思路

* 热点 key 单独处理
* 普通 key 走正常 join
* 避免所有热点都压在一个 reducer 上

## 本质

它是 **反倾斜优化策略**，不是通用 join 基型。

# 6. Multi-Way Join

这个通常不是单独的“join 算法”，更像优化方式。

如果多个表都按同一个主表 key 关联，优化器可能把多个 join 合并处理，减少中间结果和多轮 MR。相关资料里也会把它列为 join optimization 的一种。([Medium][5])

## 作用

适合：

* 星型模型
* 一个事实表连续 join 多个维表
* join key 关系比较规整

## 价值

* 减少 MR 阶段
* 减少中间数据
* 提高整体执行效率

# 7. 一张表看区别

| Join 类型                   | Join 发生位置  | 适用场景            | 主要优点               | 主要缺点            |
| ------------------------- | ---------- | --------------- | ------------------ | --------------- |
| Common / Reduce-Side Join | Reduce端    | 大表 join 大表，通用场景 | 最通用                | shuffle 成本高，易倾斜 |
| Map Join                  | Map端       | 大表 join 小表      | 快，少 shuffle        | 小表必须能放内存        |
| Bucket Map Join           | Map端       | 已分桶表 join       | 比 map join 更省资源    | 依赖 bucketing    |
| SMB Join                  | Map端 merge | 两边大表且已分桶排序      | 性能高，少 shuffle/sort | 前提最苛刻           |
| Skew Join                 | 混合优化       | key 严重倾斜        | 缓解长尾、热点 reducer    | 不是通用 join，针对性强  |

# 8. 你面试时可以怎么答

如果面试官问：

**“MapReduce/Hive 除了 skew join，还有哪些 join？区别是什么？”**

你可以这样答：

> 在 Hive/MapReduce 里，除了 skew join，常见的 join 执行策略还有 common join、map join、bucket map join 和 SMB join。
> common join 是最基础的 reduce-side join，通用性最强，但需要 shuffle，代价最大；map join 适合大表 join 小表，把小表广播到 mapper 内存里；bucket map join 是利用分桶进一步减少 map join 的代价；SMB join 适合两边大表但都已经按 join key 分桶并排序，可以在 map 端直接 merge join。
> skew join 不属于最基础的 join 类型，它主要是为了解决 join key 数据倾斜，把热点 key 单独拆开处理，避免 reducer 长尾。([hive.apache.org][1])

---

# 9. 一个记忆口诀

你可以这么记：

* **Common Join**：最普通，最重
* **Map Join**：一大一小，广播小表
* **Bucket Map Join**：分桶后更省
* **SMB Join**：分桶 + 排序，最“讲究”
* **Skew Join**：专治热点 key 倾斜
