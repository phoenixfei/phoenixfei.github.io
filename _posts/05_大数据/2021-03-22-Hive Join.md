如果只看大方向，Hive 里的 Join 可以先分成两类：

- 一类是按“Join 真正发生在什么阶段”来区分，比如 `Common Join` 和 `Map Join`
- 一类是按“在特定前提下做进一步优化”来区分，比如 `Bucket Map Join`、`SMB Join`
- 还有一类更像问题导向的优化策略，比如 `Skew Join`

所以，`Skew Join` 和前几种并不完全是同一维度的概念。前者更像“为了解决热点 key 倾斜，Hive 采取的特殊处理方案”，而后几种更像是常规 Join 的执行策略。

---

## Common Join：最通用，但代价也最大

`Common Join` 也常被叫做 `Reduce-Side Join`，它是最基础、最通用的一种 Join 方式。

它的执行流程很好理解：两边表的数据先经过 Map 阶段，按照 `join key` 输出，然后进入 Shuffle。相同 key 的记录会被分发到同一个 Reducer，最终在 Reduce 端完成关联。

也就是说，`Common Join` 的本质就是：**Join 真正发生在 Reduce 端。**

这种方式最大的优点是通用。无论两边是大表还是小表，也不要求表提前做分桶、排序，基本都能用。很多“先跑通再说”的场景，最终落地的就是这种方案。

但它的问题也非常明显：

- 需要经历完整的 Shuffle
- 网络传输、排序、落盘的开销都比较大
- 一旦某些 key 特别热，很容易让某个 Reducer 成为瓶颈

所以，`Common Join` 常常不是“最快”的方案，而是“最稳妥、最普适”的方案。

![Common Join 示意图]({{ site.baseurl }}/assets/images/hive-common-join.svg)

一个典型的 `Common Join` 写法如下：

```sql
SELECT
    o.order_id,
    o.user_id,
    p.product_name
FROM dwd_order_detail o
JOIN dim_product p
    ON o.product_id = p.product_id;
```

如果 Hive 没有把小表优化成 `Map Join`，这类普通写法通常就会走 `Reduce-Side Join`。

## Map Join：典型的大表 Join 小表

如果一边是大表，另一边是明显可以放进内存的小表，那么 Hive 很可能会选择 `Map Join`。

它的核心思路是：把小表加载到每个 Mapper 的内存里，通常组织成哈希表；然后大表正常扫描，在 Map 阶段直接完成关联。这样就不需要让两边数据都进入 Reduce，自然也就减少了大量 Shuffle 开销。

你可以把它理解成“把小表广播出去，让每个 Mapper 都随身带着一份字典”。

这类 Join 特别适合下面几种场景：

- 事实表 Join 维表
- 大表 Join 小表
- 星型模型中的维度关联

它的优势非常直接：快，而且通常快得很明显。因为少了 Reduce 侧的大量数据搬运和排序，执行性能往往优于 `Common Join`。

不过，它也有边界：

- 小表必须足够小，能够放进内存
- 如果误判了小表大小，可能会导致内存压力甚至 OOM
- 它解决不了“两边都是大表”的问题

所以，`Map Join` 的关键词不是“高级”，而是“条件合适时非常划算”。

![Map Join 示意图]({{ site.baseurl }}/assets/images/hive-map-join.svg)

在 Hive 中，可以通过提示语显式引导优化器使用 `Map Join`：

```sql
SELECT /*+ MAPJOIN(p) */
    o.order_id,
    o.user_id,
    p.product_name
FROM dwd_order_detail o
JOIN dim_product p
    ON o.product_id = p.product_id;
```

这里的 `p` 一般是小表，也就是被广播到各个 Mapper 内存中的那一侧。

## Bucket Map Join：在 Map Join 基础上进一步优化

`Bucket Map Join` 可以看成是 `Map Join` 的进一步优化。

前提是表已经按照 `join key` 做过 `bucketing`。有了这个前提之后，Hive 不一定要把整张小表广播给所有 Mapper，而是可以让大表的某个 bucket 只去匹配对应的小表 bucket。

这样做的意义在于：原本很多不必要的数据读取和广播，可以被进一步压缩。换句话说，`Map Join` 已经在减少 Shuffle 了，而 `Bucket Map Join` 则继续减少 Map 端的无效处理。

它适合的场景通常是：

- 表本身已经做了分桶
- Join key 相对稳定
- 想在 `Map Join` 的基础上继续优化资源消耗

当然，代价也很明显：

- 依赖分桶这个前置条件
- 建表和落表时就要有明确规划
- 使用条件比普通 `Map Join` 更严格

所以很多时候，`Bucket Map Join` 不是“临时救火型优化”，而是“建模阶段就设计好的优化”。

![Bucket Join 与 SMB Join 示意图]({{ site.baseurl }}/assets/images/hive-bucket-smb-join.svg)

如果希望 Hive 能利用分桶优化，通常需要在建表阶段就把分桶规则确定下来，例如：

```sql
CREATE TABLE dwd_order_bucketed (
    order_id     STRING,
    user_id      STRING,
    product_id   STRING,
    order_amount DECIMAL(10,2)
)
CLUSTERED BY (user_id) INTO 8 BUCKETS
STORED AS ORC;
```

对应的用户维表也可以按相同的关联字段分桶：

```sql
CREATE TABLE dim_user_bucketed (
    user_id    STRING,
    user_name  STRING,
    user_level STRING
)
CLUSTERED BY (user_id) INTO 8 BUCKETS
STORED AS ORC;
```

如果要让数据真正按分桶方式落表，插入数据时通常还需要结合相关设置使用。

如果表已经按 `join key` 分桶，那么对应的 Join SQL 在写法上可能依旧很普通，优化主要来自表结构本身：

```sql
SELECT
    a.user_id,
    a.order_id,
    b.user_level
FROM dwd_order_bucketed a
JOIN dim_user_bucketed b
    ON a.user_id = b.user_id;
```

真正决定它能否走 `Bucket Map Join` 的关键，不是 SQL 看起来有多特殊，而是底层表是否已经满足分桶条件。

## SMB Join：适合准备充分的大表 Join 大表

`SMB Join` 全称是 `Sort Merge Bucket Join`。

从名字就能看出，它依赖两个条件：

- `Bucket`
- `Sort`

也就是说，两张表通常都需要按 `join key` 分桶，并且在桶内有序。满足这些条件之后，对应 bucket 的数据就可以直接按有序流进行 merge，从而避免传统 `Reduce-Side Join` 那种高成本的 Shuffle 和排序。

`SMB Join` 适合的不是“随手写一条 SQL 就跑”的场景，而是下面这类情况：

- 两边都是大表
- 关联关系比较固定
- 表会被长期复用
- 前期愿意为建模规范投入成本

它的优点是：一旦前提满足，性能会比较可观，尤其适合稳定的大规模关联场景。

但它的问题也同样明显：前置要求高，建模成本高，使用门槛高。如果数据落表阶段没有按规范做好分桶和排序，后面基本就享受不到这种优化。

所以，`SMB Join` 往往不是“最常用”的，而是“准备最充分时最漂亮”的方案。

在开启相关优化配置后，满足分桶且排序条件的表可以尝试走 `SMB Join`：

```sql
SET hive.optimize.bucketmapjoin = true;
SET hive.optimize.bucketmapjoin.sortedmerge = true;
SET hive.input.format = org.apache.hadoop.hive.ql.io.BucketizedHiveInputFormat;

SELECT
    a.user_id,
    a.order_id,
    b.user_level
FROM dwd_order_sorted_bucketed a
JOIN dim_user_sorted_bucketed b
    ON a.user_id = b.user_id;
```

这类写法的重点仍然不在 SQL 本身，而在于表是否已经提前按 `join key` 做好了 `bucket + sort`。

## Skew Join：它不是新 Join 类型，而是反倾斜策略

很多人会把 `Skew Join` 和 `Map Join`、`Common Join` 并列去背，但从理解上说，最好把它单独拿出来。

`Skew Join` 要解决的问题不是“怎么 Join 更快”，而是“某些 key 太热，导致 Join 极不均衡”。

比如某个 key 的数据量远大于其他 key，那么在 `Common Join` 场景下，负责这个 key 的 Reducer 会特别慢，最后整条任务都卡在它身上，形成典型的长尾问题。

这时候 Hive 的做法通常是：

- 把倾斜 key 和普通 key 分开处理
- 普通 key 走常规 Join 逻辑
- 倾斜 key 单独采用更有针对性的方式处理
- 最后再把结果合并

所以，`Skew Join` 的本质是：**针对数据倾斜的专项优化。**

它并不替代其他 Join，而是经常依附在某种 Join 执行策略之上，对热点 key 做特殊照顾。

## 几种 Join 的核心区别

如果把几种常见 Join 放在一起看，可以这样记：

| Join 类型 | Join 发生位置 | 典型场景 | 优点 | 局限 |
| --- | --- | --- | --- | --- |
| Common Join | Reduce 端 | 大表 Join 大表、通用场景 | 通用性最强 | Shuffle 成本高，容易倾斜 |
| Map Join | Map 端 | 大表 Join 小表 | 快，Shuffle 少 | 依赖小表足够小 |
| Bucket Map Join | Map 端 | 已分桶表的优化 Join | 比 Map Join 更省资源 | 依赖 Bucketing |
| SMB Join | Map 端 Merge | 大表 Join 大表，且已分桶排序 | 性能好，减少 Shuffle/Sort | 前提严格 |
| Skew Join | 倾斜优化策略 | 热点 key 明显的场景 | 缓解长尾问题 | 不是通用 Join 类型 |

如果只用一句话概括它们的差异：

- `Common Join` 是默认通用解
- `Map Join` 是大表配小表时的高性价比解
- `Bucket Map Join` 是建立在分桶基础上的 Map 侧优化
- `SMB Join` 是对分桶和排序都要求很高的高阶方案
- `Skew Join` 是专门解决数据倾斜的问题导向型优化
