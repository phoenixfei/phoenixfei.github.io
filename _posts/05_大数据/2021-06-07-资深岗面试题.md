## 2. 资深岗拔高题

<a id="q01"></a>
### 2.1. Hadoop 技术栈里，你认为最容易被忽视的瓶颈是什么？

**答：**
我认为最容易被忽视的往往不是单个参数，而是数据组织方式本身。

比如分区设计、小文件、热点 key、指标重复计算，这些问题在数据量小时不明显，但规模一上来会比单纯资源不足更致命。

所以很多瓶颈表面看像资源问题，根上其实是数据模型和链路设计问题。

---

<a id="q02"></a>
### 2.2. 为什么很多公司不用 MapReduce 写开发了，还要考 MapReduce？

**答：**
因为 MapReduce 虽然不是主流开发框架了，但它把很多分布式计算最底层的原理讲得最清楚。

比如：

* 数据切分
* Shuffle
* 倾斜
* 容错重试
* 数据本地性

所以考 MapReduce，本质上是在考你对大数据计算基本原理的理解。

---

<a id="q03"></a>
### 2.3. 你怎么理解 Hadoop、Hive、Spark 三者的关系？

**答：**
可以把它们看成不同层次：

* Hadoop 提供分布式存储和资源管理基础设施
* Hive 提供 SQL 语义和数仓开发方式
* Spark 提供更现代的分布式计算引擎

它们不是简单替代关系，更像是在同一大数据体系里承担不同职责。

---

<a id="q04"></a>
### 2.4. 如果让你设计一个离线数仓平台，你会怎么设计底层技术栈？

**答：**
我一般会按“存储、计算、调度、治理”四层去想：

* 存储层：HDFS / 湖仓存储
* 计算层：Hive + Spark / Tez
* 调度层：YARN + 工作流调度系统
* 治理层：元数据、血缘、质量、指标平台

资深岗更看重的是你能不能把技术栈和组织治理一起考虑进去。

---

<a id="q05"></a>
### 2.5. 如果让你做一轮 Hadoop 集群容量规划，你会考虑什么？

**答：**
至少要考虑三类容量：

* 存储容量：原始数据、保留周期、副本策略
* 计算容量：峰值并发、任务时长、资源峰谷
* 增长容量：未来半年到一年数据增长预期

所以容量规划不是只看今天够不够，还要看增长趋势和故障冗余。

---

<a id="q06"></a>
### 2.6. 如果存储成本突然压力很大，你会怎么在三副本、纠删码、冷热分层之间做权衡？

**答：**
一般不会一刀切，而是分数据类型处理：

* 热数据、频繁读写数据更适合三副本
* 冷数据、归档数据更适合纠删码
* 生命周期明确的数据适合冷热分层

所以这道题的关键是：根据访问频率、恢复要求、成本敏感度做分层策略，而不是只盯着一种技术。

---

<a id="q07"></a>
### 2.7. 如果让你治理一个“跑了很多年、SQL 很乱、任务很慢”的老数仓，你会从哪里下手？

**答：**
我通常不会先全量重构，而是先抓主链路和高收益点。

常见顺序是：

1. 先找最慢、最贵、最核心的任务
2. 再统一公共口径和重复逻辑
3. 同时治理小文件、分区、资源和调度问题
4. 最后逐步推进模型和代码规范化

所以老数仓治理更像一场分阶段治理工程，不是一次性推翻重来。

---

<a id="q08"></a>
### 2.8. Hadoop 技术栈在现在企业里最大的价值还是什么？

**答：**
最大的价值仍然是：以相对成熟稳定的方式支撑大规模离线数据存储、批处理和数仓链路。

即使新技术很多，企业里仍然需要一套能稳定跑大批量任务、成本可控、生态成熟的底座，而 Hadoop 体系在这件事上依然有现实价值。

所以它的价值更多体现在“稳定生产力”，而不只是“技术新不新”。

---

<a id="q09"></a>
### 2.9. Hadoop 离线数仓链路和实时数仓链路的核心差异是什么？

**答：**
核心差异在于目标不同：

* 离线链路更重吞吐、成本和全量一致性
* 实时链路更重时效、状态管理和低延迟

所以两者在存储、计算模型、调度方式和容错策略上都会不同，但最终目标都是服务统一的数据口径。

---

<a id="q10"></a>
### 2.10. 你觉得一个资深离线数仓开发，和一个只会写 SQL 的开发，最大的区别是什么？

**答：**
最大的区别通常不在 SQL 熟练度，而在系统性思维。

资深离线数仓开发通常会同时考虑：

* 模型设计是否可演进
* 指标口径是否可治理
* 链路是否稳定、可回溯、可重跑
* 成本和性能如何平衡

所以“会写 SQL”是基础，而“能把链路做稳、把体系做清”才是资深能力。
