## 2. YARN 基础职责与面试表达

<a id="q01"></a>
### 2.1. YARN 中 ResourceManager 和 NodeManager 分别负责什么？

**答：**
在 YARN 中：

**ResourceManager** 负责全局资源调度，它站在整个集群的角度，统一管理和分配资源，决定哪些应用可以获得资源、运行在哪些节点上。

**NodeManager** 运行在每台节点上，负责本节点上的任务执行和资源管理，比如：

* 启动任务
* 监控任务状态
* 管理本机资源使用情况
* 向 ResourceManager 汇报节点状态

可以简单总结为：

* **ResourceManager：负责集群级资源调度**
* **NodeManager：负责节点级任务执行和管理**

---

<a id="q02"></a>
### 2.2. 关于 ResourceManager 和 NodeManager，面试官还可能继续追问什么？

**答：**
面试官在问完两者职责之后，通常还可能继续追问这些方向：

1. **ResourceManager 和 NodeManager 是如何协同工作的？**
   可以继续回答任务提交流程，例如应用提交后由 ResourceManager 分配资源，再由 NodeManager 在对应节点启动任务。

2. **NodeManager 挂掉了会怎么样？**
   可以回答节点故障后，ResourceManager 会感知该节点失联，并将任务重新调度到其他节点执行。

3. **ResourceManager 的高可用怎么做？**
   可以回答生产环境中通常会配置 RM HA，避免单点故障。

4. **YARN 如何处理资源过载？**
   可以从队列、资源隔离、调度策略等角度进行展开。

所以，这一块如果面试中回答得不错，往往会被继续追问到容错、调度、高可用这些更深入的内容。

---

<a id="q03"></a>
### 2.3. 如何更适合面试地回答 YARN 中 ResourceManager 和 NodeManager 的区别？

**答：**
比较适合面试的说法是：

ResourceManager 负责集群级资源调度和分配，NodeManager 负责单节点上的任务执行和资源上报。

如果再补一句职责边界，会更完整：

* ResourceManager 看的是全局，决定资源给谁
* NodeManager 看的是本机，负责 container 启动、监控和状态汇报

面试里这样回答，通常已经足够清晰。

## 3. YARN 资源调度与治理

<a id="q04"></a>
### 3.1. YARN 的整体执行流程是什么？

**答：**
一个任务提交到 YARN 后，通常会经历这几个步骤：

1. 客户端向 ResourceManager 提交应用
2. ResourceManager 先分配一个 container 启动 ApplicationMaster
3. ApplicationMaster 再向 ResourceManager 申请更多资源
4. NodeManager 在对应节点启动任务 container
5. 任务执行过程中，ApplicationMaster 负责监控、重试和状态汇总

所以 YARN 的主线就是：`提交应用 -> 启动 AM -> AM 申请资源 -> NM 启动任务`。

---

<a id="q05"></a>
### 3.2. ApplicationMaster 的作用是什么？

**答：**
ApplicationMaster 可以理解为某个应用在 YARN 上的“现场总指挥”。

它主要负责：

* 为这个应用向 ResourceManager 申请资源
* 协调各个任务 container 的启动
* 监控任务执行状态
* 失败时决定是否重试和如何恢复

所以 ResourceManager 负责全局调度，ApplicationMaster 负责单个应用内部调度。

---

<a id="q06"></a>
### 3.3. Container 是什么？

**答：**
Container 是 YARN 中资源分配和任务运行的基本单位。

它本质上描述的是“一块被分配出来的资源配额”，通常包括内存、vcore 等。任务真正运行时，就是在某个节点上的某个 container 里执行。

所以 container 不是单纯的进程概念，而是“资源 + 运行环境”的组合体。

---

<a id="q07"></a>
### 3.4. ResourceManager、NodeManager、ApplicationMaster 三者是什么关系？

**答：**
可以把三者看成三个层次：

* ResourceManager：管全局资源分配
* NodeManager：管单机资源和 container 执行
* ApplicationMaster：管单个应用的资源申请和任务协调

它们的协作关系是：RM 给资源，AM 要资源并编排任务，NM 在本机真正把任务跑起来。

---

<a id="q08"></a>
### 3.5. YARN 为什么要把资源管理和计算框架解耦？

**答：**
因为 Hadoop 不只跑 MapReduce，后面还要支持 Spark、Tez、Flink 等多种计算框架。

如果资源管理和具体计算引擎绑死，就很难在一套集群上复用资源。YARN 把资源管理抽出来后，多个计算框架都可以共享同一套底层资源池。

所以解耦的价值在于：提升资源复用能力，降低平台耦合度。

---

<a id="q09"></a>
### 3.6. YARN 支持哪些调度器？FIFO、Capacity、Fair 有什么区别？

**答：**
常见有三类：

* **FIFO**：先进先出，简单但灵活性差
* **Capacity Scheduler**：按队列容量做资源保障，适合多租户
* **Fair Scheduler**：强调公平分享资源，让作业尽量都能拿到一些资源

直观上看：

* FIFO 最简单，但容易大任务压死小任务
* Capacity 更适合企业多部门、多队列隔离
* Fair 更强调共享和公平

---

<a id="q10"></a>
### 3.7. 生产中常用哪种调度器？为什么？

**答：**
很多生产环境更常用 Capacity Scheduler。

原因通常是：

* 更适合多租户和多业务线
* 能给不同队列设置容量、上限、优先级
* 更方便做资源隔离和治理

Fair 也有场景，但从企业资源治理角度看，Capacity 往往更可控。

---

<a id="q11"></a>
### 3.8. YARN 的资源隔离是怎么做的？

**答：**
YARN 的资源隔离主要依赖 container 机制，在节点侧由 NodeManager 配合 Linux cgroups 等能力做限制。

隔离的资源通常包括：

* 内存
* CPU / vcore
* 一部分磁盘和进程级约束

它不像虚拟机那样是完全强隔离，但足以满足大多数离线计算资源治理需求。

---

<a id="q12"></a>
### 3.9. 内存和 vcore 在 YARN 中是如何管理的？

**答：**
YARN 把资源抽象成内存和 vcore 两类核心维度。任务申请资源时，需要明确要多少内存、多少 vcore。

调度时，ResourceManager 会基于节点剩余资源和队列策略决定是否分配。NodeManager 在本机负责监控 container 是否超用资源，超限时可能直接杀掉任务。

所以 YARN 的资源管理本质上就是“按配额申请，按配额调度，按配额约束”。

---

<a id="q13"></a>
### 3.10. 一个任务申请不到资源，通常有哪些原因？

**答：**
常见原因有：

* 队列容量已经满了
* 单次申请资源过大，没有节点能满足
* 集群整体资源紧张
* 队列权限、优先级或并发限制导致无法立即分配

所以排查时不要只看“有没有空闲资源”，还要看队列规则和单 container 规格是否合理。

---

<a id="q14"></a>
### 3.11. NodeManager 挂了会发生什么？

**答：**
NodeManager 挂掉后，这台机器上的 container 基本都会失效。

ResourceManager 会通过心跳超时感知节点失联，随后把这台节点上的任务视为失败。对应应用的 ApplicationMaster 会根据失败策略重新申请资源，在其他节点拉起任务。

所以节点挂掉通常影响的是局部任务，不一定直接导致整个应用失败。

---

<a id="q15"></a>
### 3.12. ResourceManager 挂了会发生什么？

**答：**
如果没有 HA，ResourceManager 挂掉会导致新资源无法分配，整个集群调度基本停摆。

已经在跑的 container 可能暂时还活着，但新的申请、调度、状态协调都会受影响。所以 RM 是典型的关键控制平面组件。

这也是为什么生产环境通常要给 RM 做 HA。

---

<a id="q16"></a>
### 3.13. YARN 的高可用是怎么实现的？

**答：**
YARN HA 通常是部署多台 ResourceManager，采用 Active / Standby 模式，再配合 ZooKeeper 做状态协调和故障转移。

当 Active RM 异常时，Standby RM 会接管调度职责，避免整个集群因单点故障停摆。

所以 YARN HA 的重点在于：主备切换和状态一致性。

---

<a id="q17"></a>
### 3.14. YARN 中任务执行失败如何恢复？

**答：**
YARN 的恢复一般分两层：

* 任务级失败：ApplicationMaster 重试单个 task / container
* 应用级失败：如果失败太多，最终应用整体失败

如果只是节点抖动、container 异常，通常 AM 会重新申请资源在其他节点重跑。

所以恢复核心在 AM，它是单应用维度的容错协调者。

---

<a id="q18"></a>
### 3.15. 为什么有时候任务本身不慢，但排队很久？

**答：**
这通常不是执行慢，而是调度慢。

常见原因包括：

* 队列资源被占满
* 高优先级任务插队
* 单次资源申请规格过大，迟迟凑不出合适节点
* 队列并发数或用户配额限制

所以这类问题要先分清：是“没资源可分”，还是“有资源但策略不让你立刻用”。

---

<a id="q19"></a>
### 3.16. 如何定位一个 YARN 任务是卡在资源申请还是执行过程？

**答：**
最直接的办法是看任务在 YARN UI 上的状态和耗时拆分。

如果长时间停留在 ACCEPTED 或资源未分配阶段，通常是卡在资源申请；如果已经 RUNNING 但某些 stage / task 很久不结束，通常是执行过程出了问题。

所以先分阶段看：等待时间长，多半是资源问题；运行时间长，多半是执行问题。

---

<a id="q20"></a>
### 3.17. YARN 中队列资源怎么规划？

**答：**
队列规划一般要结合业务优先级、任务类型和资源峰谷特征。

常见思路是：

* 核心链路和普通任务分队列
* 预留基础保障容量给关键业务
* 允许空闲资源在队列间借用，提高利用率

所以队列规划本质上是在“资源保障”和“整体利用率”之间做平衡。

---

<a id="q21"></a>
### 3.18. 如何避免一个任务把整个队列资源吃满？

**答：**
常见做法有：

* 限制单用户、单应用可使用的最大资源比例
* 给队列设置最大并发和最大容量
* 控制单个任务的 container 规格和实例数

如果治理更细一点，还会引入任务优先级和资源配额体系，避免大任务把小任务完全饿死。

---

<a id="q22"></a>
### 3.19. 离线数仓场景下，YARN 资源参数一般怎么调？

**答：**
离线数仓里，调参通常重点看这几类：

* container 内存和 vcore
* executor / task 并发数
* 队列容量和最大并发
* 应用侧并行度是否和队列资源匹配

核心原则不是“调得越大越好”，而是让单任务规格、并发度和队列总容量匹配，避免大材小用或挤爆队列。

---

<a id="q23"></a>
### 3.20. YARN 和 Kubernetes 做资源调度，有什么思路差异？

**答：**
YARN 更像是为大数据离线计算场景定制的资源调度器，天然理解 container、队列、批处理任务和 Hadoop 生态。

Kubernetes 则更通用，强调统一编排、弹性、容器生态和云原生能力。

所以差异不只是“谁更先进”，而是：

* YARN 更偏 Hadoop 计算场景优化
* Kubernetes 更偏通用平台和统一基础设施
