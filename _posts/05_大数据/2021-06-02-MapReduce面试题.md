## 2. MapReduce 核心原理

<a id="q01"></a>
### 2.1. MapReduce 中，Map 阶段和 Reduce 阶段分别主要负责什么？

**答：**
Map 阶段主要负责读取输入数据，并将输入数据转换成中间键值对。

Reduce 阶段主要负责将相同 key 的数据聚合处理，最后输出结果到 HDFS。

如果用一句话概括：

* **Map：负责映射，生成中间结果**
* **Reduce：负责聚合，生成最终结果**

这是面试中最适合先给出的简洁回答。如果面试官继续追问，再展开内部细节。

---

<a id="q02"></a>
### 2.2. MapReduce 中，Map 阶段内部的详细流程是什么？

**答：**
Map 阶段的内部流程可以概括为以下几步：

1. 从 HDFS 读取输入分片数据
2. 将输入记录解析为键值对
3. 在内存缓冲区中处理这些中间结果
4. 当缓冲区达到阈值时，触发 spill，将数据溢写到磁盘
5. spill 过程中会按照分区进行组织，并进行排序
6. 如果开启了 Map 端本地聚合，还会在这个阶段做局部聚合
7. 当多个 spill 文件生成后，还会再做一次 merge 合并

所以，Map 阶段不仅仅是“读数据和映射”，它还包含了溢写、排序、本地聚合、合并等过程。

---

<a id="q03"></a>
### 2.3. MapReduce 中，Shuffle 和 Reduce 阶段主要做了什么？

**答：**
Shuffle 阶段负责把多个 Map 输出的中间结果，按照 key 和分区规则，分发到对应的 Reduce 端。

在这个过程中，系统会完成：

* 不同 Map 输出结果的拉取
* 同一分区数据的归并
* 排序
* 将相同 key 的数据组织到一起

在 Shuffle 完成后，Reduce 阶段再接收这些已经按 key 聚集好的数据，并执行聚合逻辑，最后把结果输出到 HDFS。

所以，Shuffle 本质上是 Map 到 Reduce 之间的数据重分发和排序过程，而 Reduce 则是在这个基础上完成最终聚合。

---

<a id="q04"></a>
### 2.4. MapReduce 中，Map 端本地聚合是什么？有什么作用？

**答：**
Map 端本地聚合一般指的是 **Combiner** 机制。

它的作用是在 Map 输出结果写入磁盘或者 merge 的过程中，先对部分相同 key 的数据做一次局部聚合。这样可以减少后续 Shuffle 传输的数据量，从而减轻网络压力和 Reduce 端的负担。

它的核心价值是：

* 减少中间结果数据量
* 降低 Shuffle 开销
* 提升整体任务性能

但需要注意，不是所有场景都适合本地聚合，只有满足结合律、交换律的聚合操作才适合使用。

---

## 3. MapReduce 原理与调优

<a id="q05"></a>
### 3.1. MapReduce 的 Shuffle 为什么通常是性能瓶颈？

**答：**
Shuffle 同时涉及本地溢写、排序、网络传输、远端拉取、归并和落盘，是整个作业里最重的一段数据重分发过程。

它之所以容易成为瓶颈，主要是因为：

* 数据量大时网络传输开销高
* 排序和 merge 会消耗大量 CPU 与磁盘 IO
* Reduce 端拉取多个 Map 输出，容易出现长尾

所以很多 MapReduce 优化，本质上都是在想办法减少 Shuffle 数据量。

---

<a id="q06"></a>
### 3.2. spill 为什么会发生？spill 次数过多说明什么问题？

**答：**
spill 是因为 Map 端内存缓冲区达到阈值后，必须把中间结果先刷到磁盘，给后续数据腾空间。

spill 次数过多通常说明：

* Map 输出数据量很大
* 缓冲区偏小
* 本地聚合效果差，数据没有被提前压缩
* 单条记录或 key 分布不理想，导致 merge 成本高

spill 本身正常，但过多往往意味着 Map 端磁盘 IO 压力偏大。

---

<a id="q07"></a>
### 3.3. Combiner 为什么不能保证一定执行？

**答：**
Combiner 本质上只是 Map 端的一个可选优化，不是业务语义必须执行的一步。

执行引擎可能因为数据量、任务流程或实现细节，选择执行 0 次、1 次甚至多次 Combiner。所以程序逻辑不能依赖“Combiner 一定会执行”这个前提。

面试里最关键的一句话是：Combiner 只能优化，不能改变结果语义。

---

<a id="q08"></a>
### 3.4. 哪些场景可以用 Combiner，哪些场景不能用？

**答：**
适合用 Combiner 的，一般是满足交换律、结合律的聚合操作，比如 sum、count、max、min。

不适合直接用 Combiner 的，通常是这些场景：

* 平均值这类不能直接局部合并的逻辑
* 对顺序敏感的处理
* 会因为局部合并改变最终语义的逻辑

如果要做 average，也不是完全不能做，而是需要把中间结果设计成 `(sum, count)` 这种可合并结构。

---

<a id="q09"></a>
### 3.5. Partitioner 的作用是什么？什么时候需要自定义 Partitioner？

**答：**
Partitioner 的作用是决定每条 Map 输出记录应该发往哪个 Reduce。

默认一般按 key 的 hash 分发。如果业务上希望：

* 特定 key 落到固定 Reduce
* 某些数据分组有特殊路由规则
* 需要控制下游文件布局或二次排序配套逻辑

就会考虑自定义 Partitioner。

---

<a id="q10"></a>
### 3.6. Reducer 的数量怎么设置？过多或过少各有什么问题？

**答：**
Reducer 数量通常要结合数据量、key 分布、下游文件数量和集群资源一起看，没有固定万能值。

Reducer 太少的问题是：

* 单个 Reduce 压力大，容易慢或 OOM
* 更容易放大数据倾斜

Reducer 太多的问题是：

* 小文件增多
* 调度和启动开销变高
* 很多 Reduce 实际处理数据很少，资源利用率差

所以本质上是在并行度和管理开销之间找平衡。

---

<a id="q11"></a>
### 3.7. MapTask 的数量是怎么决定的？

**答：**
MapTask 数量主要由输入切片数决定，而切片数通常由 InputFormat、文件大小、block 大小和切片策略共同决定。

直观上可以理解成：有多少个 input split，就会有多少个 MapTask。

所以 MapTask 数量通常不是直接手动指定出来的，而是由输入数据的组织方式“切”出来的。

---

<a id="q12"></a>
### 3.8. InputFormat 的作用是什么？

**答：**
InputFormat 主要负责两件事：

* 决定输入数据如何切分成多个 split
* 为每个 split 提供对应的 RecordReader

换句话说，它既决定“切多少个 MapTask”，也决定“每个 MapTask 读数据时按什么方式解析记录”。

---

<a id="q13"></a>
### 3.9. TextInputFormat 和 CombineTextInputFormat 有什么区别？

**答：**
TextInputFormat 通常按普通文本文件切分，一个 split 对应一段输入。

CombineTextInputFormat 则更适合小文件场景，它会把多个小文件组合成一个更大的 split，减少 MapTask 数量。

所以两者最大的区别不是“都能读文本”，而是对小文件的处理策略不同。

---

<a id="q14"></a>
### 3.10. RecordReader 的作用是什么？

**答：**
RecordReader 的作用是把底层输入数据流解析成一条条逻辑记录，再交给 Mapper。

例如在 TextInputFormat 场景里，它通常会把每一行文本解析成：

* key：行起始偏移量
* value：这一行内容

所以 InputFormat 负责切分，RecordReader 负责把 split 里的数据“读成记录”。

---

<a id="q15"></a>
### 3.11. 什么是 MapReduce 的二次排序？如何实现？

**答：**
二次排序指的是：先按主 key 分组，再在同一个主 key 内部按次 key 有序。

常见实现思路是：

* 定义复合 key
* 自定义排序比较器，让主次 key 都参与排序
* 自定义分组比较器，只按主 key 分组
* 必要时配合 Partitioner，确保同一主 key 落到同一个 Reduce

这样 Reduce 收到的数据就是“按主 key 聚在一起、组内再按次 key 排好序”的。

---

<a id="q16"></a>
### 3.12. 什么是数据本地性（Data Locality）？为什么重要？

**答：**
数据本地性指的是让计算尽量靠近数据，而不是把大量数据跨网络搬运到远端再算。

它重要的原因很直接：网络通常比本地磁盘和本地内存更慢、更贵。MapTask 如果能尽量在数据所在节点或同机架节点启动，整体性能通常会更好。

所以 Hadoop 强调“移动计算比移动数据更划算”。

---

<a id="q17"></a>
### 3.13. MapReduce 中有哪些类型的数据倾斜？分别怎么处理？

**答：**
常见可以分成两类：

* **Map 侧倾斜**：比如小文件过多、切片不均，导致部分 MapTask 特别多或特别慢
* **Reduce 侧倾斜**：比如热点 key 太集中，导致个别 Reduce 特别慢

对应处理思路一般是：

* Map 侧看输入组织、合并小文件、调整切片策略
* Reduce 侧看热点 key 打散、Map Join、拆分热点数据、增加并行度

所以排查时先分清是“输入不均”还是“key 分布不均”。

---

<a id="q18"></a>
### 3.14. MapReduce 为什么适合离线批处理，不适合实时计算？

**答：**
因为 MapReduce 的执行模型有明显的阶段边界，很多中间结果要落磁盘，整个流程更偏吞吐导向而不是延迟导向。

它适合离线批处理，是因为：

* 大数据量下稳定性强
* 容错机制成熟
* 顺序扫描和批量聚合能力强

但做实时计算时，这种“多阶段、重落盘、重调度”的模型延迟太高。

---

<a id="q19"></a>
### 3.15. MapReduce 和 Spark 的核心区别是什么？

**答：**
最核心的区别是：MapReduce 更依赖磁盘中间结果，Spark 更强调以内存为中心的 DAG 计算。

因此通常表现为：

* MapReduce 更稳、更朴素，适合大批量离线处理
* Spark 在迭代计算、交互分析和复杂 DAG 上通常更快

所以很多公司虽然不用 MapReduce 做主开发了，但仍然会考它，因为很多分布式计算基本原理都能从它看清楚。

---

<a id="q20"></a>
### 3.16. 为什么说 MapReduce 是磁盘型计算框架？

**答：**
因为 MapReduce 在多个阶段之间，往往依赖本地磁盘或 HDFS 落地中间结果，而不是尽量全程驻留内存。

这带来的好处是容错简单、恢复稳定，坏处是 IO 开销大、延迟高。

所以说它是“磁盘型计算框架”，并不是说它不用内存，而是说它的核心执行路径非常依赖磁盘。

---

<a id="q21"></a>
### 3.17. 一个 Reduce 很慢，如何判断是倾斜还是资源不足？

**答：**
先看这个慢 Reduce 处理的数据量和 key 分布。如果它输入记录数远高于其他 Reduce，通常更像数据倾斜。

如果各个 Reduce 输入量差不多，但慢任务机器的 CPU、内存、磁盘或网络明显有问题，更像资源不足或节点异常。

实践里一般会一起看：

* Reduce 输入量是否极不均衡
* 热点 key 是否集中
* 机器资源、GC、磁盘 IO 是否异常

---

<a id="q22"></a>
### 3.18. MapReduce 任务失败后会怎么重试？

**答：**
MapReduce 会先在任务级别做重试，也就是某个 task attempt 失败后，调度器会在其他节点重新拉起新的 attempt。

如果某台机器连续导致失败过多，还有可能被标记为不可靠节点，减少后续调度。

如果失败超过可容忍次数，最终才会上升到 job 失败。

---

<a id="q23"></a>
### 3.19. 推测执行（Speculative Execution）是什么？适合什么场景？

**答：**
推测执行是指：系统发现某个任务明显比同类任务慢时，会额外启动一个备份实例，谁先跑完就采用谁的结果。

它适合的场景通常是：

* 集群里偶发慢节点较多
* 任务本身是纯计算型、无副作用
* 长尾主要来自节点抖动而不是业务数据倾斜

所以它更像“对抗偶发慢机”的办法，不是治倾斜的根方案。

---

<a id="q24"></a>
### 3.20. 什么时候应该关闭推测执行？

**答：**
这些场景通常要考虑关闭：

* 任务会向外部系统写数据，重复执行有副作用
* 长尾根因是数据倾斜，而不是节点慢
* 集群资源已经紧张，不想再启动额外副本任务

因为推测执行会额外占资源，如果场景不合适，反而可能让整体更慢。
