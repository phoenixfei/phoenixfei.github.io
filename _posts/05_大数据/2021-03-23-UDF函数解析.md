在 Hive 与 Spark 的日常开发中，内置函数已经能够覆盖大部分字符串处理、日期处理、JSON 解析与聚合计算场景。但在以下场景中，仍然需要引入自定义函数：

- 业务规则无法直接用 SQL 表达
- 同一套逻辑需要在多个任务之间复用
- 需要将复杂处理封装成统一的数据口径
- 需要把一行输入展开为多行或多列结果

本文按照输入输出模型展开说明，重点回答三个问题：`UDF`、`UDAF`、`UDTF` 分别解决什么问题；Hive 中为什么经常看到 `lateral view + explode`；Spark 与 Hive 在函数扩展方式上有哪些差异。

---

## 1. 自定义函数概览

从输入输出关系看，三类函数的差异非常清晰：

![UDF、UDAF 与 UDTF 对比示意图]({{ site.baseurl }}/assets/images/hive-spark-udf-overview.svg)

| 类型 | 全称 | 输入输出特征 | 典型用途 | 代表写法 |
| --- | --- | --- | --- | --- |
| `UDF` | User Defined Function | 一行输入，一个值输出 | 字段清洗、脱敏、映射、规则判断 | `evaluate()` / `spark.udf.register()` |
| `UDAF` | User Defined Aggregate Function | 多行输入，一个值输出 | 自定义统计、评分、聚合标签 | evaluator / `UserDefinedAggregateFunction` |
| `UDTF` | User Defined Table-Generating Function | 一行输入，多行或多列输出 | 数组展开、多值字段拆分、结构化解析 | `forward()` / `explode()` |

可以将三者概括为一句话：

**`UDF` 是“一进一出”，`UDAF` 是“多进一出”，`UDTF` 是“一进多出”。**

在选型时，通常可按以下顺序判断：

1. 如果只是对单个字段做转换，优先考虑 `UDF`。
2. 如果需要对一组记录进行汇总计算，优先考虑 `UDAF`。
3. 如果需要把一行拆成多行或多列，优先考虑 `UDTF` 或内置的 `explode`。
4. 如果内置函数已经足够表达业务逻辑，应优先使用内置函数，而不是立即开发自定义函数。

## 2. Hive 函数开发

### UDF：单行单值

`UDF` 最常见，也最容易落地。它的实现方式通常是继承 `org.apache.hadoop.hive.ql.exec.UDF`，然后编写 `evaluate()` 方法。

常见场景包括：

- 手机号或身份证脱敏
- 编码标准化
- 日期格式转换
- 业务标签映射
- 复杂条件判断封装

下面以“手机号脱敏”为例：

```java
package com.example.hive.udf;

import org.apache.hadoop.hive.ql.exec.UDF;
import org.apache.hadoop.io.Text;

public class MaskPhoneUDF extends UDF {

    public Text evaluate(Text phone) {
        if (phone == null) {
            return null;
        }

        String value = phone.toString().trim();
        if (value.length() != 11) {
            return new Text(value);
        }

        String masked = value.substring(0, 3) + "****" + value.substring(7);
        return new Text(masked);
    }
}
```

完成函数开发后，下一步就是将其注册到 Hive 中。

### Hive UDF 注册

Hive 中常见的注册方式有两类：**临时注册** 与 **长期注册**。两者的主要区别如下：

| 注册方式 | 生效范围 | 生命周期 | 典型场景 |
| --- | --- | --- | --- |
| 临时注册 | 当前会话 | 会话结束后失效 | 调试、自测、单次任务 |
| 长期注册 | 当前数据库或指定数据库 | 持久保存在 metastore 中 | 平台公共函数、团队复用函数 |

**临时注册**

临时注册的特点是简单直接，通常适合开发验证或一次性任务。常见步骤如下：

1. 使用 `ADD JAR` 将自定义函数的 `jar` 包加入当前会话的 classpath。
2. 使用 `CREATE TEMPORARY FUNCTION` 在当前会话中注册函数。
3. 在当前会话内直接调用该函数。

```sql
add jar /data/udf/my-hive-udf.jar;

create temporary function mask_phone
as 'com.example.hive.udf.MaskPhoneUDF';

select
    phone,
    mask_phone(phone) as masked_phone
from user_info;
```

这种方式的限制也很明确：会话结束后需要重新执行注册语句。

**长期注册**

如果函数需要被多个任务或多个会话长期复用，更适合注册为长期函数。长期函数会写入 metastore，因此不需要每次登录 Hive 后重复执行 `CREATE TEMPORARY FUNCTION`。

```sql
create function dwd.mask_phone
as 'com.example.hive.udf.MaskPhoneUDF'
using jar 'hdfs:///udf/my-hive-udf.jar';
```

对于长期注册，需要注意以下几点：

1. 函数会注册到当前数据库，或者显式注册到指定数据库，例如 `dwd.mask_phone`。
2. `USING JAR` 可以在函数首次被引用时自动加载依赖资源。
3. 在非本地模式下，`jar` 路径通常应放在 HDFS 等共享存储中，而不是本地磁盘路径。

如果只是想快速验证逻辑，优先使用临时注册；如果希望沉淀为团队共用能力，更适合使用长期注册。

### UDAF：聚合计算

`UDAF` 的核心不是“写一个最终结果”，而是“设计一套可在分布式环境中逐步合并的中间状态”。因此，理解 `UDAF` 时应重点关注下面四个阶段：

1. `iterate`：逐条处理输入记录。
2. `terminatePartial`：输出局部聚合结果。
3. `merge`：合并局部聚合结果。
4. `terminate`：返回最终结果。

下面以“按组计算字符串最大长度”为例，展示 `GenericUDAFEvaluator` 的核心实现：

```java
public static class MaxStringLengthEvaluator extends GenericUDAFEvaluator {

    static class MaxLenAggBuffer implements AggregationBuffer {
        int maxLen;
    }

    @Override
    public AggregationBuffer getNewAggregationBuffer() {
        MaxLenAggBuffer buffer = new MaxLenAggBuffer();
        buffer.maxLen = 0;
        return buffer;
    }

    @Override
    public void reset(AggregationBuffer agg) {
        ((MaxLenAggBuffer) agg).maxLen = 0;
    }

    @Override
    public void iterate(AggregationBuffer agg, Object[] parameters) {
        if (parameters == null || parameters.length == 0 || parameters[0] == null) {
            return;
        }

        MaxLenAggBuffer buffer = (MaxLenAggBuffer) agg;
        int currentLen = parameters[0].toString().length();
        buffer.maxLen = Math.max(buffer.maxLen, currentLen);
    }

    @Override
    public Object terminatePartial(AggregationBuffer agg) {
        return ((MaxLenAggBuffer) agg).maxLen;
    }

    @Override
    public void merge(AggregationBuffer agg, Object partial) {
        if (partial == null) {
            return;
        }

        MaxLenAggBuffer buffer = (MaxLenAggBuffer) agg;
        int partialMax = Integer.parseInt(partial.toString());
        buffer.maxLen = Math.max(buffer.maxLen, partialMax);
    }

    @Override
    public Object terminate(AggregationBuffer agg) {
        return ((MaxLenAggBuffer) agg).maxLen;
    }
}
```

对应 SQL 用法如下：

```sql
add jar /data/udf/my-hive-udaf.jar;

create temporary function max_str_len
as 'com.example.hive.udaf.MaxStringLengthUDAF';

select
    group_id,
    max_str_len(keyword) as max_len
from search_log
group by group_id;
```

如果需要理解 `terminatePartial`，可以把它看成“输出可被后续阶段继续合并的中间结果”，而不是“提前输出最终答案”。例如 `avg` 的中间状态通常不是平均值，而是 `(sum, count)`。

### UDTF：一行多出

`UDTF` 与普通函数最大的区别在于：它返回的不是单个值，而是一张结果表。因此，开发时通常需要关注以下三个方法：

1. `initialize()`：定义输出列结构。
2. `process()`：处理当前输入行。
3. `forward()`：把结果逐条发出。

下面以“将逗号分隔字符串拆成多行”为例：

```java
package com.example.hive.udtf;

import java.util.ArrayList;
import java.util.List;
import org.apache.hadoop.hive.ql.exec.UDTF;
import org.apache.hadoop.hive.serde2.objectinspector.ObjectInspector;
import org.apache.hadoop.hive.serde2.objectinspector.ObjectInspectorFactory;
import org.apache.hadoop.hive.serde2.objectinspector.StructObjectInspector;
import org.apache.hadoop.hive.serde2.objectinspector.primitive.PrimitiveObjectInspectorFactory;

public class SplitToRowsUDTF extends UDTF {

    @Override
    public StructObjectInspector initialize(StructObjectInspector argOIs) {
        List<String> fieldNames = new ArrayList<>();
        List<ObjectInspector> fieldOIs = new ArrayList<>();

        fieldNames.add("tag");
        fieldOIs.add(PrimitiveObjectInspectorFactory.javaStringObjectInspector);

        return ObjectInspectorFactory.getStandardStructObjectInspector(fieldNames, fieldOIs);
    }

    @Override
    public void process(Object[] args) throws org.apache.hadoop.hive.ql.metadata.HiveException {
        if (args == null || args.length == 0 || args[0] == null) {
            return;
        }

        String[] arr = args[0].toString().split(",");
        for (String item : arr) {
            forward(new Object[]{item.trim()});
        }
    }

    @Override
    public void close() {
    }
}
```

SQL 注册与调用方式如下：

```sql
add jar /data/udf/my-hive-udtf.jar;

create temporary function split_to_rows
as 'com.example.hive.udtf.SplitToRowsUDTF';

select
    user_id,
    tag
from user_profile
lateral view split_to_rows(tags) t as tag;
```

## 3. `lateral view` 与 `explode`

`explode` 在 Hive 中本质上可以视为一种 `UDTF`。它的作用是把数组、`map` 或多值字段展开为多行结果，而 `lateral view` 的作用是把这些展开后的结果重新与原始行关联起来。

![`lateral view` 与 `explode` 协作示意图]({{ site.baseurl }}/assets/images/hive-lateral-view-explode.svg)

可以用下面这组三步来理解执行过程：

1. 先对当前行执行 `split(tags, ',')`，把字符串转成数组。
2. 再对数组执行 `explode()`，把一个数组元素拆成多行。
3. 最后通过 `lateral view` 保留原行中的 `user_id` 等字段，并将其与拆分后的结果一起输出。

示例数据如下：

| user_id | tags |
| --- | --- |
| 1001 | sports,movie,music |
| 1002 | game,tech |

对应 SQL 为：

```sql
select
    user_id,
    tag
from user_profile
lateral view explode(split(tags, ',')) t as tag;
```

输出结果如下：

| user_id | tag |
| --- | --- |
| 1001 | sports |
| 1001 | movie |
| 1001 | music |
| 1002 | game |
| 1002 | tech |

如果希望在输入字段为空时尽量保留原表记录，可以使用 `lateral view outer`：

```sql
select
    user_id,
    tag
from user_profile
lateral view outer explode(split(tags, ',')) t as tag;
```

这里需要特别注意一点：

**是否需要 `lateral view`，关键不在于函数是不是“自定义函数”，而在于它是不是“会把一行变成多行的表生成函数”。**

## 4. Spark 函数用法

相比 Hive，Spark 的函数扩展方式更偏向“在作业内注册，再通过 SQL 或 DataFrame API 使用”。

### Spark UDF

下面以 PySpark 为例，实现与 Hive 中相同的手机号脱敏逻辑：

```python
from pyspark.sql import SparkSession
from pyspark.sql.types import StringType

spark = SparkSession.builder.appName("SparkUDFDemo").getOrCreate()

def mask_phone(phone):
    if phone is None:
        return None
    if len(phone) != 11:
        return phone
    return phone[:3] + "****" + phone[7:]

spark.udf.register("mask_phone", mask_phone, StringType())

spark.sql("""
select
    phone,
    mask_phone(phone) as masked_phone
from user_info
""").show()
```

如果使用 DataFrame API，也可以直接写成：

```python
from pyspark.sql import functions as F

mask_phone_udf = F.udf(mask_phone, StringType())

df2 = df.withColumn("masked_phone", mask_phone_udf(F.col("phone")))
```

### Spark UDAF

Spark 中常见的自定义聚合方式有两种：

- 继承 `UserDefinedAggregateFunction`
- 使用更推荐的强类型 `Aggregator`

下面给出 `UserDefinedAggregateFunction` 的核心骨架：

```scala
import org.apache.spark.sql.Row
import org.apache.spark.sql.expressions.MutableAggregationBuffer
import org.apache.spark.sql.expressions.UserDefinedAggregateFunction
import org.apache.spark.sql.types._

class MaxStringLengthUDAF extends UserDefinedAggregateFunction {

  override def inputSchema: StructType =
    StructType(StructField("value", StringType) :: Nil)

  override def bufferSchema: StructType =
    StructType(StructField("maxLen", IntegerType) :: Nil)

  override def dataType: DataType = IntegerType

  override def deterministic: Boolean = true

  override def initialize(buffer: MutableAggregationBuffer): Unit = {
    buffer(0) = 0
  }

  override def update(buffer: MutableAggregationBuffer, input: Row): Unit = {
    if (!input.isNullAt(0)) {
      buffer(0) = math.max(buffer.getInt(0), input.getString(0).length)
    }
  }

  override def merge(buffer1: MutableAggregationBuffer, buffer2: Row): Unit = {
    buffer1(0) = math.max(buffer1.getInt(0), buffer2.getInt(0))
  }

  override def evaluate(buffer: Row): Any = buffer.getInt(0)
}
```

注册与调用方式如下：

```scala
val spark = SparkSession.builder().appName("SparkUDAF").getOrCreate()

spark.udf.register("max_str_len", new MaxStringLengthUDAF)

spark.sql("""
  select
    group_id,
    max_str_len(keyword) as max_len
  from search_log
  group by group_id
""").show()
```

### Spark `explode`

Spark 中的 `explode` 功能与 Hive 接近，但语法通常更直接。SQL 写法如下：

```sql
select
    user_id,
    explode(split(tags, ',')) as tag
from user_profile;
```

DataFrame 写法如下：

```python
from pyspark.sql import functions as F

df2 = df.select(
    "user_id",
    F.explode(F.split(F.col("tags"), ",")).alias("tag")
)
```

如果字段类型为 `map`，也可以展开为键值对：

```python
df2 = df.select(
    "user_id",
    F.explode(F.col("props")).alias("k", "v")
)
```

Spark 中有一个非常实用的经验原则：

**能使用内置函数时，优先使用内置函数；只有在内置表达能力不足时，再考虑 UDF 或 UDAF。**

## 5. Hive 与 Spark 对比

Hive 与 Spark 都支持函数扩展，但它们的使用习惯和工程位置并不完全相同。

| 维度 | Hive | Spark |
| --- | --- | --- |
| 函数定位 | SQL 引擎扩展 | 作业内函数扩展 |
| 常见开发语言 | Java | Scala / Java / Python |
| UDF 使用方式 | 打包 `jar` 后注册 | 作业内注册后调用 |
| UDAF 开发重点 | evaluator 与中间状态设计 | buffer 与聚合模型设计 |
| `explode` 常见写法 | `lateral view explode(...)` | `select explode(...)` |
| 更典型的使用场景 | 平台公共函数、离线数仓 | 作业内逻辑封装、DataFrame 处理 |

在实际项目中，可以按下面的思路判断：

1. 如果函数需要在 Hive 平台侧长期复用，更适合沉淀为 Hive 自定义函数。
2. 如果逻辑只在某个 Spark 作业内使用，通常直接在 Spark 中注册函数更灵活。
3. 如果需求可以用 `split`、`explode`、`sum`、`max` 等内置函数完成，优先使用内置函数。
4. 如果主要目标是提升 SQL 可读性、统一业务口径，再考虑开发自定义函数。

## 6. 开发注意事项

无论在 Hive 还是 Spark 中，自定义函数的稳定性都高度依赖实现细节。开发时建议重点关注以下事项：

- **空值处理**：明确 `null` 输入的行为，避免在生产环境中出现非预期异常。
- **类型一致性**：尤其是 `UDAF`，要同时考虑输入类型、中间状态类型与最终输出类型。
- **性能边界**：Spark 内置函数通常优于 JVM UDF，JVM UDF 通常优于 Python UDF。
- **副作用控制**：不建议在函数内部访问远程服务、数据库或文件系统，以免放大分布式执行成本。
- **版本兼容**：Hive 与 Spark 的依赖版本应尽量与集群保持一致，避免 `ClassNotFoundException` 或 `NoSuchMethodError`。
- **可维护性**：对于需要长期复用的规则，应优先将逻辑封装为命名明确、输入输出清晰的公共函数。

最后可以用一句话概括本文内容：

**`UDF` 负责字段级变换，`UDAF` 负责聚合级计算，`UDTF` 负责结果级展开；在 Hive 中，`lateral view` 是承接展开结果的重要语法，而在 Spark 中，同类能力通常以更直接的 API 形式出现。**
